如何打破模型必须串行生成token的魔咒，并行产生后k个token？
我们可以训练 k 个辅助模型，直接额预测 prompt 后面 2 到 k 个位置的token。那么，辅助模型和原始模型可以独立运行，从而并行生成后 k 个token。

while True：

  1. predict : 使用 base model 预测下一个token，其他的辅助模型预测下 2-k 个token （ 并行计算一步，一次forward时间
  2. verify ：使用 base model 计算生成 token 的 logits（ 并行计算一步，一次forward时间
  3. accept ：验证结果的token和Predict阶段预测的token相同则保留。如果不同，则后面的token预测都错误。

predict 和 verify 各自需要1次 forward 的时间，所以 forward 的调用次数从 m 减少到 2m/k 。

于是进一步优化，在 Verify 时顺便做了下一个token的预测。
这样 Predict 和 Verify 阶段可以合并，验证同时也获得了后 k 个 token 的候选。
这使得除了第一次迭代，每次迭代只需调用一次模型 forward ，而不是两次，从而将解码所需的模型调用次数减半。
进一步将模型调用次数从 2m/k 减少到 m/k + 1 。

  1. predict ：还是一样的，并行计算出下面 k 个token。（ 仅仅在开始并行计算一步，一次forward时间

    这里生成的 k 个 token，并不一定最准确。一般随着 k 增大，越往后被生成的 token 越不准确，所以才需要下面的 verify 。

while True：

  1. verify ：并行对所有生成的 token 进行verify，在 verify 的同时会生成后面 k 个候选 token 。

    这里作者通过修改模型结构，让模型能够同时完成 verify 和 next predict 的过程

  2. 如果存在 token logits 不一致，则直接放弃这个 token ( bus )，直接使用上面已经生成的token作为确定的 答案，使用其生成的候选 token 作为下一步骤 verify 的对象。

模型架构：

在原始模型的 the decoder output 和 the final projection layer 之间插入一个FFN，将输入 [batch_s, seq, hidden_s]\text{[batch\_s, seq, hidden\_s]}[batch_s, seq, hidden_s] 转化为 [batch_s, seq, k* hidden_s]\text{[batch\_s, seq, k* hidden\_s]}[batch_s, seq, k* hidden_s] ，然后分割为 [batch_s, seq, k, hidden_s]\text{[batch\_s, seq, k, hidden\_s]}[batch_s, seq, k, hidden_s] 表示输出 k 个 next token 。最后加一个残差连接。

近似推理 ：

按道理说，Verify 阶段保留更多 token，速度越快

所以可以考虑放宽 accept 的条件，不一定是严格按照 greedy 的方法（token概率最大）。

作者也提供了一些优化技巧，它们可以加速推理速度，但是会导致结果和自回归采样不一致。

  1. Top-k Selection：只要 token 概率值为 top-k 内，则接受
  2. Distance-Based Selection：和最优token的距离在超参数 ϵ\epsilonϵ 内
  3. Minimum Block Size：在模型的 predict token 中，第一个 token 肯定是可以被接受，但是后面的token，往往会被删除。有时候会出现每一次仅仅保留一个 token 的情况。所以设置最短保留长度 l≤kl\le kl≤k 。

训练 FFN ：

  1. Frozen Parameters：将原始模型参数冻结，只更新新加入的FFN层参数。
  2. Finetuning：以原始参数为初始化值对全部参数进行微调，但在最终性能上可能会有所损失。
  3. Distillation：蒸馏很适合并行解码，蒸馏数据是原始模型用相同的超参数但不同的随机种子进行beam search产生的。

投机采样 speculative decoding

  1. 就是使用一个小模型来做草稿，然后使用大模型做纠正检查。
  2. 小模型的参数量要远小于原模型参数量一个级别才效果明显。
  3. 小模型和原模型的tokenizer最好一模一样，不然会增加额外的解码、编码时间。

过程：

给定原始输入 prompt token 序列（蓝色序列

  1. 首先使用小模型/辅助模型生成 γ=5\gamma = 5γ=5 个新的token（红色序列
  2. 将蓝色序列和红色序列拼接在一起，放入原始模型中，然后在红色的对应位置，通过并行计算，一步计算所有结果（绿色序列
  3. 然后从前向后对比红色和绿色序列，只要有一个错了，那后面就不能要了。于是从这个位置开始使用原始大模型进行序列生成。

分析：

对于自回归序列模型，一步错，步步错。因此，虽然后面的小模型预测了一个340（红色）和大模型预测的340（绿色是一样的），但是因为前面的 token 不一致，所以后面的 token 完全不能用。

虽然小模型预测了5步，全都错了，但是也不亏。因为小模型的5步的计算时间，远远小于大模型一次预测的时间。

大模型直接利用原始token（蓝色序列），要预测新的5个token，计算起码需要跑5次。先使用小模型预测5个试一试，然后大模型借助矩阵计算的并行特性，一次性就可以验证这5个中，前几个都是对的。如果有对的，那节约的时间可不是一点点（因为小模型远小于大模型，所以小模型消耗的时间基本可以忽略不记）。

Break the Sequential Dependency of LLM Inference Using Lookahead Decoding | LMSYS Org

和投机采样比，最大的优势是 不需要draft model ，减少了依赖。

每一 decoding step 包含了两个并行的步骤：the lookahead branch & the verification branch

  1. The Lookahead branch : maintains a fixed-sized, 2D window to generate n-grams

    两个超参数维护这个 2D 窗口

    window size WWW 控制窗口的长度，即多少个 Guess Token 稍后进行 parallel decoding

    N-gram size NNN 控制在生成 n-gram 轨迹时，需要看多少个历史的 step

    对于每一个token，根据包含自己的 N-1 个前缀去生成下一个 token，以此组成了 N-gram

  2. The verification branch : selects and verifies promising n-gram candidates
