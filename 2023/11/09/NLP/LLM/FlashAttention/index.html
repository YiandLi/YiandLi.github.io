显存受限

在现代GPU中，计算速度已经远超过了显存访问速度，transformer中的大部分计算操作的瓶颈是显存访问。
对于显存受限的操作，IO感知是非常重要的，因为显存读写占用了大部分的运行时间。

为什么呢？来具体分析下。

  • 计算带宽（math bandwidth） 指的是处理器每秒钟可以执行的数学计算次数，单位通常是OPS(即operations/second)。如果用浮点数进行计算，单位是FLOPS。

  • 内存带宽 （memory bandwidth） 指的是处理器每秒钟从内存中读取的数据量，单位是bytes/second。

在计算科学中，给定硬件，找出一个操作的性能瓶颈是非常重要的。
一个操作的性能瓶颈有两类：计算受限（math-bound）和内存受限（memory-bound）。

从数学上分析，浮点数计算次数为 NopN_{o p}Nop​ ，内存访问量为 $N_{b y t e } $ 计算带宽为 $ B W_{m a t h} $ ，内存带宽为 $ B W_{m e m} $ 。

访问内存花费的时间为 TmemT_{m e m}Tmem​ ，计算花费的时间为 TmathT_{m a t h}Tmath​ 。

Tmem =Nbyte BWmem Tmath =NopBWmath \begin{aligned} T_{\text {mem }} & =\frac{N_{\text {byte }}}{B W_{\text {mem }}} \\ T_{\text {math }} & =\frac{N_{o p}}{B W_{\text {math }}} \end{aligned} Tmem ​Tmath ​​=BWmem ​Nbyte ​​=BWmath ​Nop​​​

由于可以"边计算上一个，边读/写下一个"，访问内存和计算的时间可以重叠，此时总的运行时间为 max⁡(Tmem ,Tmath )\max \left(T_{\text {mem }}, T_{\text {math }}\right)max(Tmem ​,Tmath ​) 。

当 Tmath>TmemT_{math}>T_{m e m}Tmath​>Tmem​ 时，硬件性能受限于计算带宽，为计算受限 (math-bound)。在数学上，下面的不等式成立：

Tmath >Tmem →NopNbyte >BWmath BWmem T_{\text {math }}>T_{\text {mem }} \quad \rightarrow \quad \frac{N_{o p}}{N_{\text {byte }}}>\frac{B W_{\text {math }}}{B W_{\text {mem }}} Tmath ​>Tmem ​→Nbyte ​Nop​​>BWmem ​BWmath ​​

反之，当 $ T_{math}<T_{m e m}$ 时，硬件性能受限于内存带宽，为内存受限 (memory-bound)。在数学上，下面的不等式成立：

Tmath <Tmem →Nop Nbyte <BWmath BWmem T_{\text {math }}<T_{\text {mem }} \quad \rightarrow \quad \frac{N_{\text {op }}}{N_{\text {byte }}}<\frac{B W_{\text {math }}}{B W_{\text {mem }}} Tmath ​<Tmem ​→Nbyte ​Nop ​​<BWmem ​BWmath ​​

通常将 ops/bytes =NopNbyte =\frac{N_{o p}}{N_{\text {byte }}}=Nbyte ​Nop​​ 定义为算术强度（arithmetic intensity）。给定硬件的情况下，对于一个模型或算法，若算术强度 >BWmath BWmem >\frac{B W_{\text {math }}}{B W_{\text {mem }}}>BWmem ​BWmath ​​ ，则为计算受限；反之为内存受限。
假如用float16进行计算，A100-40GB SXM的 BWmath BWmem =312×10121555×109≈201\frac{B W_{\text {math }}}{B W_{\text {mem }}}=\frac{312 \times 10^{12}}{1555 \times 10^9} \approx 201BWmem ​BWmath ​​=1555×109312×1012​≈201 flops /bytes 。若一个操作的算术强度 ops/types >201 ，此时的性能受限于计算带宽；反之，性能受限于内存带宽。

Attention 中：矩阵乘是计算受限，而 softmax ，dropout，mask，reduction操作 等是内存受限。

所以：内存受限 --> 减少HBM读写次数 --> kernel融合 --> 满足SRAM的内存大小 --> 分块计算

FlashAttention

GPU 组成：

  1. SRAM（Static Random-Access Memory）：计算实际工作区域

    读写带宽速度很快，19TB/S ； 但是容量很小 20MB

  2. HBM（High Bandwidth Memory）：A100-40GB版本里的HBM就是显存40GB / nvidia-smi

    读写带宽速度很慢，1.5TB/S ；但是容量很大

Soft-max Forward 流程：

S=QK⊤∈RN×N,P=softmax⁡(S)∈RN×N,O=PV∈RN×d\mathbf{S}=\mathbf{Q} \mathbf{K}^{\top} \in \mathbb{R}^{N \times N}, \quad \mathbf{P}=\operatorname{softmax}(\mathbf{S}) \in \mathbb{R}^{N \times N}, \quad \mathbf{O}=\mathbf{P V} \in \mathbb{R}^{N \times d} S=QK⊤∈RN×N,P=softmax(S)∈RN×N,O=PV∈RN×d

S,P\mathbf S, \mathbf PS,P 矩阵都是很大的，需要在HBM中实例化来进行存储，这样就会带来很多HBM的访问次数。

需要从HBM中读取5次，写入HBM3次。是十分耗时的；所以本文想要减少交换次数。

即：

  1. 算子融合，减少读写HBM的次数：从输入 Q,K,V\mathbf{Q}, \mathbf{K}, \mathbf{V}Q,K,V 到输出 O\mathbf{O}O 的整个过程进行融合，以避免 S,P\mathbf{S}, \mathbf{P}S,P 矩阵的存储开销，减少显存占用，实现端到端的延迟缩减。

  2. 分块 tiling ，在不访问整个输入的情况下计算 softmax ，减少显存占用：为了让计算过程的结果完全在SRAM中，摆脱对HBM的依赖，可以采用分片操作，每次进行部分计算，确保这些计算结果能在SRAM内进行交互，待得到对应的结果后再进行输出。

分块 Tiling 计算

SRAM的内存大小有限，不可能一次性计算完整的注意力，因此必须进行分块计算，使得分块计算需要的内存不超过SRAM的大小。否则会导致OOM。

注意力机制的计算过程是 矩阵乘法 --> scale --> mask --> softmax --> dropout --> 矩阵乘法 ，矩阵乘法和逐点操作（scale，mask，dropout）的分块计算是容易实现的，

重计算

为了后向传递计算梯度，前向计算时通常需要将某些中间结果写回到HBM中，这会产生额外的HBM读写次数，减慢运行时间。

传统 Softmax 需要保留 S ，P 中间矩阵；而 Flash Attention没有为后向传递保存很大的中间结果矩阵，而仅仅保留了 m;lm ; lm;l 两个统计量。

后向传递时通过分块的方式重新计算注意力矩阵 SSS ，进而得到概率值 PPP ：P=exp(S−m)/lP = exp(S-m)/lP=exp(S−m)/l 。

利用GPU非匀称的存储器层次结构，实现了显著的内存节省（从平方增加转为线性增加）和计算加速（提速2-4倍），而且计算结果保持一致。

Parallelism — 在序列长度维度上并行化

一般输入部分的维度为： (batch_size, head_num, seq_length, hidden_size//head_num)

此前并行是在 batch_size 和 head_num 层面上进行的，也就是一个 grid 上有 batch_size * head_num 个 block ， 每个 block 被分给一个 SM 。

A100里有108个SM， 4090有128个SM ； 假如这个数字比较小，比如 batch_size 很小的时候，可能就会导致运行效率低下。

因此可以加入在一个序列上的并行，同一个 head 里的一个序列可以拆给多个SM处理。

外层循环为 Q ，可以 thread blocks 上进行并行计算，彼此不需要进行通信。同时，在 batch dimension，heads dimension，sequence length 维度上进行并行，增加了GPU占用率。

最终，Outer Loop 为 Q ，每一个 batch 的 token 的 head 的 Q 的工作分配给一个 CUDA thread block ，完成和所有 K，V 块的运算 。

Work Partitioning Between Warps

上一节讲到，分配 batch_size * head_num 个 block，这一节讲如何分配 warp 。

在 FlashAttention 中，将 KV 切割为 4 个 warp，在外循环中使用不同的 warp 。每次内部循环都需要从HBM频繁读写 OOO。这种方式被称为“split-K”方案，是非常低效的。

在 FlashAttention2 中，将 Q 切割为 4 个 warp，在外循环中使用，而 KV 作为一个整体进行共享。内部循环直接在SRAM上计算出 OOO ，不需要读写，效率高。

优化内外循环读写流程

• Flash Attention 1 最大的问题在于O[i]需要频繁的读写SRAM 缓存

• Flash Attention 2 将Q当作外循环，KV当作内循环， 将O[i]的在一个Q[i]周期算完

同时，warps之间不再需要相互通信去处理 QiQ_iQi​ ，所以外循环可以放在不同的 thread block 上。这个交换的优化方法是由 Phil Tillet 在 Triton[17] 提出并实现的。

讨论：

  1. v1中，o需要不断的从hbm读写，但是，v2中，kv也要反复从hbm里面读到sram在这一点上，算打个平手。

    v2 Work Partitioning Between Warps这一部分主要的改进是，warp之间的通信，同步大大减少，导致性能提高。

  2. v1算法O是先load到share memory，但是由于在kv维度划分warp，需要不同同步后做reduce操作。而v2的做法reduce（说白了就没有reduce了）直接发生在tensor core，效率提高很多。

  3. v1计算该块的O 时，每个warp算出了矩阵乘的部分结果，需要先写入共享内存。然后再同步，并重新读取，reduce成一个。会有额外的共享内存读写开销，和不同warp同步的开销。
    （v1外循环KV的话，每个block得到的局部O,大小同最终O,需要同步才能得到最终结果）
    v2对Q分块，每个warp能得到矩阵乘的部分完整结果。可以直接写到共享内存中该结果的对应部分，不需要warp之间同步了。也不需要读取共享内存进行更新。

Flash Decoding

大模型推理加速之Flash Decoding：更小子任务提升并行度

问题：

训练时， FlashAttention 对 batch size 和 query / sequence length 的维度上进行了并行优化，也就是一共有 batch size * query / sequence length 个 block 。

但是在推理时， query length 一般为 1 ，因为是 token by token 进行预测 ，导致Decoding可并行的维度减少。

这时候，batch size = block num 。如果 batch size 小于GPU上的SM数量（例如A100上有108个SMs），那么整个计算过程只使用了GPU的一小部分！

Flash Attention 1 中的操作为：

Outer Loop 为 Q ，每一个 batch 的 token 的 head 的 Q 的工作分配给一个 CUDA thread block ，完成和所有 K，V 块的运算 。

但是这里 Q 数量仅仅为 1，导致 block 数量不足。

上面提到FlashAttention对batch size和query length进行了并行化加速，Flash-Decoding在此基础上增加了一个新的并行化维度：keys/values的序列长度。

主要包含以下三个步骤（可以结合上图来看）：

  1. 将keys和values分成较小的block（沿着inner loop：Key & Value 切成5分）
  2. 使用FlashAttention并行计算query与每个block的注意力（这是和FlashAttention最大的区别）。对于每个block的每行（因为一行是一个特征维度），Flash Decoding会额外记录attention values的log-sum-exp（标量值，用于第3步进行rescale）
  3. 对所有output blocks进行reduction得到最终的output，需要用 log-sum-exp 值来重新调整每个块的贡献

实际应用中，第1步中的数据分块不涉及GPU操作（因为不需要在物理上分开），只需要对第2步和第3步执行单独的kernels。虽然最终的reduction操作会引入一些额外的计算，但在总体上，Flash-Decoding通过增加并行化的方式取得了更高的效率。

flashattention之前是维护了一个全局的O，来合并分块结果，这样似乎就是顺序合并了。

正常分治算法应该像归并排序一样，两两并行合并，而不是和一个全局的去合并。但是感觉GPU上实现比较困难吧，从动态图来看，flash-decoding 第一层合并是并行的，reduce阶段第2层合并还是顺序的。
