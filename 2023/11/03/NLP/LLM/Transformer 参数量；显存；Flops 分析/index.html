  • 显存分析命令 ：

    不要使用 nvidia-smi ，因为 PyTorch是有缓存区的设置的，意思就是一个 Tensor 就算被释放了，进程也不会把空闲出来的显存还给 GPU，而是等待下一个 Tensor 来填入这一片被释放的空间 。所以不能准确地给出某一个时间点具体的 Tensor 占用的显存，而是显示的已经分配到的显存缓冲区和 Torch 在创建cuda进程时所需开销的和，也就是 reserved_memory 和 torch context 显存之和。

    torch.cuda.memory_allocated() ：精准地反馈当前进程中 torch.Tensor 所占用的 GPU 显存 (注意是只包括torch.Tensor)

    torch.cuda.max_memory_allocated() ：告诉我们到调用函数为止所达到的最大的显存占用字节数

  • Cuda Context

    https://discuss.pytorch.org/t/how-do-i-create-torch-tensor-without-any-wasted-storage-space-baggage/131134

    PyTorch will create the CUDA context in the very first CUDA operation, which can use ~600-1000MB of GPU memory depending on the CUDA version as well as the used device. PyTorch itself will allocate the needed memory and will use an internal cache mechanism.

  • 训练过程显存分析：

    模型参数、优化器状态和参数梯度是始终在占用着存储空间的，中间激活值在反向传播之后就自动被清空了。

    定义模型：模型参数 * 4 （ float32 每个参数占用 4 bytes ）

    前向传播：

      input 在输入模型前需要计算显存

  然后生成激活值显存，显存增加等于每一层模型产生的结果的显存之和，且跟batch_size成正比。

    反向传播：

      主要是为了求 loss

  后向传播会将模型的中间激活值给消耗并释放掉；然后留下梯度占用的激活值。

  在第一次执行时，会首先分配特定的空间，用于存储梯度。

  然后这部分空间基本就不变了，即便模型进行更新，也不会变化，会等待用于存储下一批梯度。

    参数更新：

      第一次执行（`.step()`）时，会为每一个参数初始化其优化器状态，对于这里的AdamW而言，每一个参数需要额外的4*2=8个字节。 

  第二次开始，不会再额外分配显存。

  由于计算机计算的特性，有一些计算操作在计算过程中是会带来额外的显存开销的。但是这种开销在torch.memory_allocated中是不能被察觉的。

  比如在AdamW在进行某一层的更新的时候，会带来2倍该层参数量大小的临时额外开销。这个在max_memory_allocated中可以看到。

float16 比float32小一半，显存占用可以降低一半；减少网络通信的开销；硬件针对float16有优化，速度更快；

但是，float16最大的问题是"下溢"。模型的更新通常是 gradient×lr\text{gradient}×\text{lr}gradient×lr，随着模型的训练，这个值往往会很小，可能会超出float16表示的精度。

导致模型权重都不再更新，模型难以收敛。

所以使用混合精度训练，模型权重、梯度使用float16，优化器参数为float32。此外，优化器还需要保存一份float32版本的权重。

混合精度的具体过程如下：

  1. 使用float16权重进行前向传播；
  2. 反向传播得到float16的梯度；
  3. 通过优化器计算出float32精度的权重更新量；
  4. 更新float32权重；
  5. 将float32权重转换为float16；

  • 训练的显存消耗：

    4大部分的显存开销

    模型参数(parameters)，模型参数的梯度(gradients)，优化器状态(optimizer states)，中间激活值(intermediate activations)

    1. 主要的显存消耗

      假设有一个参数量为 Ψ\PsiΨ 的模型，并使用 Aadm 作为优化器。

      模型的参数和梯度使用 float16, 那么显存的消耗分别是 2Ψ2 \Psi2Ψ 和 2Ψ2 \Psi2Ψ 。

      Aadm会维护一个float32的模型副本, 则会消耗 4Ψ4 \Psi4Ψ。此外, 根据上面介绍的Aadm优化器, Adam需要为每个参数维护两个状态变量 v,rv,rv,r 。
      由于 v,rv,rv,r 均是 float 32 , 所以显存占用则为 4Ψ+4Ψ4 \Psi+4 \Psi4Ψ+4Ψ 。

      模型会消耗 2Ψ+2Ψ=4Ψ2 \Psi+2 \Psi=4 \Psi2Ψ+2Ψ=4Ψ 的显存, Aadm优化器则消耗 4Ψ+4Ψ+4Ψ=12Ψ4 \Psi+4 \Psi+4 \Psi=12 \Psi4Ψ+4Ψ+4Ψ=12Ψ 的显存。最终, 总的显存消耗为 16Ψ16 \Psi16Ψ。（这里没考虑 float32 gradint，也就是优化器得到的梯度）

      • 总的来说：优化器状态 4Ψ+4Ψ=8Ψ4 \Psi+4 \Psi = 8\Psi4Ψ+4Ψ=8Ψ ；模型权重（优化器部分 4Ψ4 \Psi4Ψ + 半精度部分 2Ψ2\Psi2Ψ = 6Ψ6 \Psi6Ψ ；梯度部分 半精度梯度 2Ψ2 \Psi2Ψ （可以释放），给到优化器部分 4Ψ4 \Psi4Ψ ，一共18Ψ/20Ψ18 \Psi/ 20 \Psi18Ψ/20Ψ ，如果不计算给优化器的 4Ψ4 \Psi4Ψ 则为 16Ψ16 \Psi16Ψ 。

    2. 剩余的显存消耗

      1. 激活(Activations)

        除了模型参数、梯度、优化器状态外，占用显存的大头就是前向传递过程中计算得到的中间激活值了，需要保存中间激活以便在后向传递计算梯度时使用。

        激活值指的是：
        前向传递过程中计算得到的，并在后向传递过程中计算图上需要用到的所有张量。应该也可以理解成计算图的构建和保存需要的显存。
        这里的激活不包含模型参数和优化器状态，但包含了dropout操作需要用到的mask矩阵。

        表示每一层module产生的结果的显存之，在训练中也会消耗大量的显存。

        • 激活显存分析

          激活（activations）：前向传递过程中计算得到的，并在后向传递过程中需要用到的所有张量。

          这里的激活不包含模型参数和优化器状态，但包含了dropout操作需要用到的mask矩阵。

          在分析中间激活的显存占用时，只考虑激活占用显存的大头，忽略掉一些小的buffers。

          在分析中间激活的显存占用时，假设中间激活值是以float16或bfloat16数据格式来保存的，每个元素占了2个bytes。

          LayerNorm

          用到层的输入，输入维度为 [b,s,h][b,s,h][b,s,h] ，用到均值 μ\muμ 和方差 σ2\sigma^2σ2 ，维度为 [b,s][b,s][b,s] 。

          通常 hhh 很大，所以 [b,s][b,s][b,s] 可以被忽略，LN 激活元素数量也就近似估计为 bshbshbsh ，显存占用为 2bsh2bsh2bsh 。

          Self-Attention

          1. QKVQKVQKV 矩阵：输入部分元素个数 bshbshbsh ，显存占用 2bsh2bsh2bsh 。
          2. QKQKQK ：输入部分为两个 bshbshbsh ，显存占用 2∗2bsh=4bsh2*2bsh = 4bsh2∗2bsh=4bsh 。
          3. Softmax\text{Softmax}Softmax ：输入为 [b,a,s,s][b,a,s,s][b,a,s,s] ，占用显存大小为 2bas22bas^22bas2 ，aaa 为注意力头数 。
          4. 计算完 Softmax，需要加一个 dropout 操作，输入仍然为 [b,a,s,s][b,a,s,s][b,a,s,s] ，占用显存大小为 bas2bas^2bas2 。
            注意：dropout操作的mask矩阵，每个元素只占1个bytes。
          5. 使用 attention score 在 value 上 进行聚合，attention score 输入和上一步操作一致，value 输入大小为 bshbshbsh，一共占用了 2bas2+2bsh2bas^2 + 2bsh2bas2+2bsh 。
          6. 计算输出映射，也就是乘以 O 矩阵，输入为 bshbshbsh ，占用显存大小 2bsh2bsh2bsh 。
          7. Droput，占用显存大小 bshbshbsh 。

          综上，self-attention 占用显存大小为 11bsh+5bas211bsh + 5bas^211bsh+5bas2

          MLP

          第一层输入 bshbshbsh ，占用显存 2bsh2bsh2bsh

          激活函数输入 4bsh4bsh4bsh ，占用显存 8bsh8bsh8bsh

          第二层输入 4bsh4bsh4bsh，占用显存 8bsh8bsh8bsh

          最后 dropout 输入 bshbshbsh，占用显存 bshbshbsh

          综上，一共占用了 19bsh19bsh19bsh 显存。

          Embedding and Output Layer

          embedding 层：不需要中间激活

          Output 层：输入为 bsh，占用显存为 2bsh2bsh2bsh

          总之

          所以一层 layer 占用了 2∗2bsh+(11bsh+5bas2)+(19bsh)=34bsh+5bas22*2bsh + (11bsh + 5bas^2) + (19bsh) = 34bsh + 5bas^22∗2bsh+(11bsh+5bas2)+(19bsh)=34bsh+5bas2 ， lll 层就占用了 l×(34bsh+5bas2)l \times (34bsh + 5bas^2)l×(34bsh+5bas2) 显存 。

          如果层数很多，可以忽略 output layer 显存，总体近似为 l×(34bsh+5bas2)l \times (34bsh + 5bas^2)l×(34bsh+5bas2) 显存 。

        降低这部分显存的方案：Activation checkpointing(或者activation re-computation) 该方法以33%的重计算为代价，将激活的显存占用减少至总激活的均方根。

      2. 临时缓存区(Temporary buffers)

        用于存储中间结果的临时buffer，例如在all-reduce时，需要一个平坦的buffer来融合所有的梯度，从而改善吞吐量。

      3. 显存碎片

        即使在有足够显存的情况下，也可能会导致Out of Memory，这是由于显存碎片导致的。
        在进程发出显存请求时，如果没有连续的显存来满足请求，即使总的显存仍然足够，该请求也会失败。

  • 对比中间激活与模型参数的显存大小

    中间激活显存大小为：l×(34bsh+5bas2)l \times (34bsh + 5bas^2)l×(34bsh+5bas2) ，和输入数据量呈现正相关。随着批次大小 bbb 和序列长度 sss 的增大，中间激活占用的显存会同步增大。

    模型参数显存大小为：2×12lh2=24lh22\times 12 l h^2= 24 l h^22×12lh2=24lh2 ，和输入数据量无关

    优化器状态占用的显存大小也是与模型参数量有关，但与输入数据的大小无关。

    当我们训练神经网络遇到显存不足OOM（Out Of Memory）问题时，通常会尝试减小批次大小来避免显存不足的问题，这种方式减少的其实是中间激活占用的显存，而不是模型参数、梯度和优化器的显存。

    通常会采用 激活重计算 技术来减少中间激活，理论上可以将中间激活显存从 O(n)O(n)O(n) 减少到 O(n)O(\sqrt n)O(n​) ，代价是增加了一次额外前向计算的时间，本质上是“时间换空间”。

FLOPs

FLOPs，floating point operations，表示浮点数运算次数，衡量了计算量的大小 。

对于 R1×n⋅Rn×1\mathbb R^{1\times n} \cdot \mathbb R^{n\times 1}R1×n⋅Rn×1 ，计算需要 nnn 次乘法操作，n−1n-1n−1 次加法操作，记得还有个 bias，就是 2n2n2n FLOPs 。

对于 [a,b,c]⋅[c,d,e]=[a,b,d,e][ a,b,c] \cdot [c,d,e] = [ a,b,d,e][a,b,c]⋅[c,d,e]=[a,b,d,e] ，计算为相乘维度的两倍2c2c2c 乘以其他维度 abdeabdeabde ，即 2abcde2abcde2abcde 。

计算量与参数量的关联

当隐藏维度 hhh 比较大，且远大于序列长度 sss 时，我们可以忽略一次项，计算量可以近似为 4bshV+l(24bsh2+4hbs2)∼24bsh2∗l4bshV + l(24bsh^2+ 4hbs^2 ) \sim 24bsh^2∗l4bshV+l(24bsh2+4hbs2)∼24bsh2∗l 。

前面提到了参数量为 12lh212 l h^212lh2 ，token 数量为 bsbsbs ，因为 24bsh2×l12lh2×bs=2\dfrac{24bsh^2\times l }{12 l h^2\times bs}=212lh2×bs24bsh2×l​=2 ；
所以 在一次前向传递中，对于每个token，每个模型参数，需要进行2次浮点数运算，即一次乘法法运算和一次加法运算。

一次训练迭代包含了前向传递和后向传递，后向传递的计算量是前向传递的 222 倍 。

因此，一次训练迭代中，对于每个token，每个模型参数，需要进行 2∗(1+2)=62∗(1+2)=62∗(1+2)=6 次浮点数运算。

训练时间估计

模型参数量和训练总tokens数决定了训练transformer模型需要的计算量。

给定硬件GPU类型的情况下，可以估计所需要的训练时间。

训练时间（也就是GPU算完这么多flops的计算时间）不仅跟GPU类型有关，还与GPU利用率有关。

计算端到端训练的GPU利用率时，不仅要考虑前向传递和后向传递的计算时间，还要考虑 CPU/GPU加载数据、优化器更新、多卡通信和记录日志的时间。

关于加载数据，最慢的是 GPU 的时间，数据导入范围为：Main Memory → Cache → GPU 。

一般来讲，GPU利用率一般在 0.3∼0.55 之间 。

对于每个token，每个模型参数，前向传播需要 2 倍FLOPs，反向传播需要 4 倍FLOPs 。

使用激活重计算技术来减少中间激活显存需要进行一次额外的前向传递，因此前向传递 + 后向传递 + 激活值重计算的系数 2+4+2=82+4+2= 82+4+2=8 FLOPs 。

推理时间估计

N 为 模型参数量
prefill 阶段，s 比较大，计算时间 >> 搬运时间，所以为计算密集型。
decode 阶段，s = 1，计算时间 << 搬运时间，所以为传输密集型。

Latency：延迟 ，即从输入到输出最后一个 token 的延迟。
Latency = (TTFT) + (TPOT) * (the number of tokens to be generated). Latency 可以转换为 Tokens Per Second (TPS)：TPS = (the number of tokens to be generated) / Latency。

Throughput：吞吐量 ，即每秒针处理的请求数 / 生成的token数量。
Prefill 阶段在比较小 Batch Size 下可以获得比较大的计算强度，相应的吞吐也很高；而 Decoding 阶段需要比较大的 Batch Size 才能获得相对高的计算强度及吞吐

关注点：
在线流式应用： 对 TTFT、TPOT、Latency 敏感，需要尽可能快的生成 token。
离线批量应用： 对 Throughput 敏感，需要在单位时间内尽可能多的生成 token。

推理性能卡点

一般增加 batch size 可以有效增加吞吐量，但是

  1. kv cache 线性增加
  2. 带宽瓶颈

解决：

  1. 算子融合
  2. 降低 KV cache 大小 -> 量化 / 压缩 / vllm
  3. 模型量化 -> 降低模型需要的显存，降低带宽使用率
  4. Flash attention
