Abstract and Introduction

将「事件类型」和「论元角色」转化为「自然语言 Query」的形式检索句子中的每个token，来判定并且抽取出「触发词」和对应「论元」。

Event Centered Sentiment Analysis

Posted on 2022-07-17 Edited on 2022-07-18

论文：Implicit Sentiment Analysis with Event-Centered Text Representation

研究领域为「Implicit sentiment analysis」，即没有「显示情感词」出现的场景下的「情感分析/分类」；否则直接字典匹配了。
本文中，作者探究了「event-centric implicit sentiment analysis」，即如何利用已有的「sentiment-aware event」进行「情感分析/分类」。
作者将事件信息定义为一个三元组，「事件类型」+ <subject, predicate, object>（类似关系抽取，没有论元分类），目标是「情感分析/分类」。

主要内容：

  • 作者认为当前的方法对于事件信息的利用都过于简单或者过于间接，所以提出了一种更加直接的编码方式，对事件进行编码，来更有效地帮助「情感分析/分类」。
    提出的新模型使用了 hierarchical tensor-based composition 机制（个人感觉就是 hierarchical bi-affine）。
  • 此外，作者提出了一个 给定事件信息的隐式情感分类数据集：EveSA：Impli-Sentiment-Analysis。

比如：
Sentence: ‘You abandon me for a week to go off on holiday with daddy, come back and barely 2 days later you go off out with him again.’
Event triplet: <you, abandon, me>
Event type: abandonment
Sentiment: negative

Dataset: EveSA

在 FrameNet 和 Tweet数据 上进行过滤，人工筛选，详见论文。

Model

主要分为三部分

  1. 使用「hierarchical tensor-based composition」将事件元组信息 < subject, predicate, object > 和句子进行整合，得到新的表征。
  2. 除了「情感分类损失 LsL_sLs​ 」外，使用「事件类型分类损失 LeL_eLe​ 」进行联合训练，让模型保留事件知识。
  3. 考虑到相同类型的事件在不同的句子中有不同的三元组表达形式，作者认为「句子表征 hsenh_{\mathrm{sen}}hsen​」应该服从「事件表征 eee」的高斯分布，即 hsen∼N(e,σ2)h_{\mathrm{sen}} \sim \mathcal{N}\left(e, \sigma^{2}\right)hsen​∼N(e,σ2) ，让「句子表征空间」和「事件表征空间」更加接近。
    为此，使用一个「高斯相似度损失 LsimL_{sim}Lsim​ 」进行限制，和上述两个损失函数同时做联合训练。


事件三元组和句子是同一个 bert 编码，三元组经过 Hierarchical Tensor-based Compositions 机制得到一个事件三元组表征 eee ；句子则直接得到句子表征 hsenh_{sen}hsen​ 。

2021IEEE 事件抽取综述

Posted on 2022-06-28 Edited on 2022-06-29

论文：A Compact Survey on Event Extraction: Approaches and Applications

Abstract

  • Task definition, paradigm, and models of event extraction.
  • Numerous methods, datasets, and evaluation metrics focusing on deep learning-based models

Introduction

定义

  • 事件抽取（Event Extraction）是信息抽取的子任务（Information Extraction）
  • 事件（Event）的定义：可以被表示为一个「状态的转换」，包含时间，地点，行为和参与者等。
  • 事件抽取（Event Extraction）是指从非结构化文本中抽取出结构化形式的定义，包含 “who, when, where, what, why, how” 等描述信息。
  • 依托于 NER，Semantic Parsing，Relation Extraction 等技术

一些术语/内容

  • Event mentions：文档中，表示事件的一个短语或者句子，包含触发词和论元。
  • Entity and Entity type
  • Event trigger
  • Event argument
  • Argument role

实现方式

  1. Event detection task：
    1. 触发词识别(trigger identification)：触发词是事件的核心，表示着事件的存在。
    2. 触发词分类(trigger classification)：判断事件类型
  2. Argument extraction task
    1. 论元识别(argument identification)
    2. 论元角色分类(argument role classification)：使用 (trigger, argument)/(argument1, argument2) 对判断论元类型。

分类

Domain

  • Close-domain event extraction：事件类型属于一个特定的数据集
  • Open-domain event extraction：事件被看作一类 topic 的集合，常使用分类或者聚类的方法。
    • 得到属于同一个topic的一系列事件

Manner and method

可以简化成多个分类任务

  • Pipeline paradigm

    • 首先抽取出事件类型，然后根据特定的 事件模版(event schema) 抽取论元。
    • 方法：序列标注，阅读理解，sequence-to-structure 生成 …

  • Joint paradigm

    • 一次性判别出事件，论元的类型
    • 防止误差传递

Classification-based Task

首先定义所有 event type 和 argument role

  1. 先根据 event mention 确定 event type
  2. 判断每个 argument 是否属于目标事件的某个 argument role

Machine Reading Comprehension-based Task

首先定义所有 event type 和 argument role，然后给每个 argument role 定义一个 问题模版 Question schema

  1. 先根据 event mention 确定 event type
  2. 遍历每个 argument role，使用对应问题模版让 MRC模型 输出对应 argument

Sequence Labelling-based Task

可以直接标注出论元类型

Sequence-to-structure Generation-based Task

使用一种端到端的方式一次性抽取出所有（事件信息，）论元信息

Scenarios

  • Document-level Event Extraction Scenario
  • Open-domain Event Extraction Scenario
  • Low-Resource Event Extraction Scenario
  • Multilingual Event Extraction Scenario
  • Chinese Event Extraction Scenario

EVENT EXTRACTION PARADIGM

Pipeline-based Paradigm


简化为多个单独的分类任务
如果trigger判别错误，则后续全部错误，所以这种范式将trigger看作事件的核心（需要精度高的trigger判别模型）。

  • 缺点：误差传播 和 如何划分模型
  • 优势：
    • 灵活性高
    • 后面的model可以借鉴前一层model给出的判别信息，即增加一些特征（using the information of triggers）。
  • paper related：
    • [39]DMCNN：“Event extraction via dynamic multi-pooling convolutional neural networks,” in ACL, 2015.
      ：two dynamic multi-pooling convolutional neural networks for trigger classification and argument classification.
    • [37]PLMEE：“Exploring pre-trained language models for event extraction and generation,” in ACL, 2019.
    • [41]MRC-based：“Event extraction by answering (almost) natural questions,” in EMNLP, 2020, 2020.

Joint-based Paradigm

可以同时考虑 trigger 和 argument，但是目前(2021)不如 pipeline 效果好。
避免误差传播，所以此种范式下 trigger 和 argument 同样重要，但是就不能在提取 argument 的时候使用 trigger 信息了。

  • 方法：

    1. 首先找出所有的候选实体（不区分trigger和argument）
    2. 然后使用两个分类模型同时判别trigger和argument

  • 使用的特征：

    • local features：
      • text sequence features
      • local window features
    • global features：
      • between different event triggers
      • between different event arguments
      • between event triggers and event arguments

这样构造一个存储累积事件概率数组时间复杂度为 O(N)O(N)O(N) ，每次线性查找的时间复杂度为 O(N)O(N)O(N) 。

Naive方法

while True 直到所有列面积都为 1 ：

  1. 找出其中面积小于等于 1 的列 i ，列 i 是一定要被填充的，所以在 Prob[i] 中填上其现有面积。
  2. 从面积大于 1 的列中，选出一个可以填满 i 列的列 j ，更新其剩余值Prob[j]，同时设置Alias[i]=j

算法复杂度为 O(n2)O(n^2)O(n2)

更快的构建方法

一个更好的办法是用两个队列A,B去储存面积大于1的index，和小于1的index。
每次从两个队列中各取一个a,b，用a填充b，b出队列B，
再看a减去补给之后，如果大于1，继续放入A中，如果等于1，则也出去，如果小于1则放入B中。

该算法每次总能稳定地pop一个概率索引，同时单次操作复杂度都是 O(1)O(1)O(1) ，所以算法复杂度为 O(n)O(n)O(n) 。

事件抽取简介

Posted on 2022-06-25 Edited on 2022-06-28
  • 限定域事件抽取 Close-domain Information Extraction：使用预定义的事件模式从文本中发现和提取所需的特定类型的事，事件模式包含多个事件类型及其相应的事件结构。
  • 开放域事件抽取 Open-domain Information Extraction：构建开放域事件抽取不拘泥于精确地将事件具体要素进行精确抽取，其主要目的一般是通过聚类、文本语义分割等无监督手段，在开放的文本数据中分析、检测出事件，以供后续的分析。

四个子任务:

  • 触发词检测
  • 事件/触发词类型识别
  • 事件论元检测
  • 论元角色识别

事件抽取任务可以由两个步骤主要组成：

  • 事件检测(Event Detection)：根据上下文识别出触发词以及代表的事件类型
  • 事件元素识别(Argument Detection)：事件元素/论元 是指事件的参与者。根据所属的事件模版，抽取相应的论元，并为其标上正确的元素标签。

分类

总体可以分为两个大类：元事件抽取和主题事件抽取。

元事件抽取

表示一个动作的发生或状态的变化，往往由动词驱动，也可以由能表示动作的名词等其他词性的词来触发。

特征范围

  • 句子级的事件抽取方法：只使用句子内部获取的特征，是所有事件抽取方法通用的特征。
  • 篇章级的事件抽取方法：包含了跨句、跨文档抽取的特征信息，是面向实际任务挖掘的特效特征。

学习方式

  • Pipeline：触发词检测识别和论元检测识别的顺序执行
  • Joint：将几种任务联合执行，同时得到结果。

主题事件抽取

关键：如何描述同一个主题事件的文档集合，以及如何通过篇章内或跨篇章的理解技术将这些集合中分散的主题事件片段进行归并。
研究中通常采用 事件框架 或 本体 来表示主题事件的基本组成以及各成分间的联系。

介绍

transformers/v4.2.2/training

Transformers 提供了 Trainer 和 TFTrainer 两个训练器接口，支持 train，fine-tune，eval 任何的 Transformers 模型。

  • Trainer 同样支持自定义模型，但是模型 forward() 方法输出的第一项必须是用于优化的损失。
  • Trainer 使用内置默认的 collate 方法整合 batch，并进行处理(collat_fn)；如需要，可以使用 data_collator 参数传递自己的 collator 函数。用于从dataset 中拿到数据，并且处理成模型需要的输入形式。
  • 可以自定义 compute_metrics 方法，得到 eval 的结果，最终结存储在 logging_dir 文件夹下。

class transformers.TrainingArguments

需要用到 TrainingArguments类 初始化 Trainer类。

初始化参数

只选择一些重要的

  • output_dir (str)：模型预测结果和保存模型的地址。
  • logging_dir (str, optional)：默认runs/**CURRENT_DATETIME_HOSTNAME**。
  • overwrite_output_dir (bool, optional, defaults to False)：如果设置True，则会重写output_dir中的内容。如果设置False并且output_dir指向一个checkpoint，则会继续训练。
  • evaluation_strategy (str or EvaluationStrategy, optional, defaults to "no")：设置训练时，eval的策略：
    • 'no'：不 evaluate
    • 'steps'：每eval_steps进行 eval 和 log。
    • 'epoch'：每个epoch结束后进行 eval 和 log。
  • logging_steps (int, optional, defaults to 500)
  • eval_steps (int, optional)：默认和 logging_steps一致。
  • save_steps (int, optional, defaults to 500)
  • save_total_limit (int, optional)：存储 checkpoints 的数量，删除旧的。
  • prediction_loss_only (bool, optional, defaults to False)：当 eval 和 predict 时，只返回 loss。
  • per_device_train_batch_size (int, optional, defaults to 8)
  • per_device_eval_batch_size (int, optional, defaults to 8)
  • dataloader_drop_last (bool, optional, defaults to False)：是否删除最后一个不完整的 batch。
  • gradient_accumulation_steps (int, optional, defaults to 1)：backward/update，logging，evaluating，saving 都生效。
  • learning_rate (float, optional, defaults to 5e-5)
  • weight_decay (float, optional, defaults to 0)
  • max_grad_norm (float, optional, defaults to 1.0)
  • num_train_epochs (float, optional, defaults to 3.0)
  • max_steps (int, optional, defaults to -1)：如果设置，则num_train_epochs失效。
  • lr_scheduler_type (str or SchedulerType, optional, defaults to "linear")详见SchedulerType
  • (int, optional, defaults to 0)：线性，从0到learning_rate。
  • no_cuda (bool, optional, defaults to False)：当cuda可用时，是否使用。
  • seed (int, optional, defaults to 42)
  • fp16 (bool, optional, defaults to False)
  • local_rank (int, optional, defaults to -1)
  • load_best_model_at_end (bool, optional, defaults to False)：如果设置True，则save_step失效，模型会在每次 eval 后被存储。
  • metric_for_best_model (str, optional)：和load_best_model_at_end一起使用，默认为 loss 值，注意设置greater_is_better参数。
  • greater_is_better (bool, optional)
  • label_smoothing_factor (float, optional, defaults to 0.0)
