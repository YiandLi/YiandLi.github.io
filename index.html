Trainer

Posted on 2022-05-29 Edited on 2022-05-31

介绍

transformers/v4.2.2/training

Transformers 提供了 Trainer 和 TFTrainer 两个训练器接口，支持 train，fine-tune，eval 任何的 Transformers 模型。

  • Trainer 同样支持自定义模型，但是模型 forward() 方法输出的第一项必须是用于优化的损失。
  • Trainer 使用内置默认的 collate 方法整合 batch，并进行处理(collat_fn)；如需要，可以使用 data_collator 参数传递自己的 collator 函数。用于从dataset 中拿到数据，并且处理成模型需要的输入形式。
  • 可以自定义 compute_metrics 方法，得到 eval 的结果，最终结存储在 logging_dir 文件夹下。

class transformers.TrainingArguments

需要用到 TrainingArguments类 初始化 Trainer类。

初始化参数

只选择一些重要的

  • output_dir (str)：模型预测结果和保存模型的地址。
  • logging_dir (str, optional)：默认runs/**CURRENT_DATETIME_HOSTNAME**。
  • overwrite_output_dir (bool, optional, defaults to False)：如果设置True，则会重写output_dir中的内容。如果设置False并且output_dir指向一个checkpoint，则会继续训练。
  • evaluation_strategy (str or EvaluationStrategy, optional, defaults to "no")：设置训练时，eval的策略：
    • 'no'：不 evaluate
    • 'steps'：每eval_steps进行 eval 和 log。
    • 'epoch'：每个epoch结束后进行 eval 和 log。
  • logging_steps (int, optional, defaults to 500)
  • eval_steps (int, optional)：默认和 logging_steps一致。
  • save_steps (int, optional, defaults to 500)
  • save_total_limit (int, optional)：存储 checkpoints 的数量，删除旧的。
  • prediction_loss_only (bool, optional, defaults to False)：当 eval 和 predict 时，只返回 loss。
  • per_device_train_batch_size (int, optional, defaults to 8)
  • per_device_eval_batch_size (int, optional, defaults to 8)
  • dataloader_drop_last (bool, optional, defaults to False)：是否删除最后一个不完整的 batch。
  • gradient_accumulation_steps (int, optional, defaults to 1)：backward/update，logging，evaluating，saving 都生效。
  • learning_rate (float, optional, defaults to 5e-5)
  • weight_decay (float, optional, defaults to 0)
  • max_grad_norm (float, optional, defaults to 1.0)
  • num_train_epochs (float, optional, defaults to 3.0)
  • max_steps (int, optional, defaults to -1)：如果设置，则num_train_epochs失效。
  • lr_scheduler_type (str or SchedulerType, optional, defaults to "linear")详见SchedulerType
  • (int, optional, defaults to 0)：线性，从0到learning_rate。
  • no_cuda (bool, optional, defaults to False)：当cuda可用时，是否使用。
  • seed (int, optional, defaults to 42)
  • fp16 (bool, optional, defaults to False)
  • local_rank (int, optional, defaults to -1)
  • load_best_model_at_end (bool, optional, defaults to False)：如果设置True，则save_step失效，模型会在每次 eval 后被存储。
  • metric_for_best_model (str, optional)：和load_best_model_at_end一起使用，默认为 loss 值，注意设置greater_is_better参数。
  • greater_is_better (bool, optional)
  • label_smoothing_factor (float, optional, defaults to 0.0)

* / ** 传入参数用法

在变量前添加单个星号或两个星号，实现多参数的传入或变量的拆解

单星号 *

用于对 列表list 或 元组tuple 中的元素进行取出（unpack）。
对一个普通变量使用单星号前缀，实际上是对变量的一次拆解操作，将变量中单独的元素拆解出来，然后依次传入函数。

双星号 **

双星号 ** 会将字典首先转换成关键字参数的形式，就相当于使用字典中的键作为变量名。
注意如果 字典中的键和函数不一致（多一个或者少一个），会报错。

* / ** 接收参数用法

单星号 *

该位置接受任意多个非关键字（non-keyword）参数，在函数中将其转化为元组

双星号 **

该位置接受任意多个关键字（keyword）参数，在函数**位置上转化为词典

即函数内部可以使用b[paramter1]的方式进行调用，挺鸡肋的。

P-tunning v2

Posted on 2022-05-14 Edited on 2022-05-22

论文：P-Tuning v2: Prompt Tuning Can Be Comparable to Fine-tuning Universally Across Scales and Tasks
代码：P-tuning v2

Abstract and Introduction

将 NLU 任务分为两类

  • simple classification tasks
  • hard sequence labeling tasks：NER，抽取式问答…

微调的弊端：

  • 在训练过程中：很耗费内存，因为必须存储所有参数的梯度和优化器状态。
  • 在推理过程中：要为每个任务保留一份模型参数

P-tunning v1

  • 冻结模型参数，只训练「continuous prompt」；减少了内存（存储参数的梯度和优化器状态）。
  • 缺点：Lack of Universality
    • Lack of universality across scales.
      使用正常大小的预训练模型时（<10 billion），效果不如 fine-tuning。只有使用 10-billion 量级的模型时，在 simple tasks 上的表现才能和 fine-tuning 碰一碰。
    • Lack of universality across tasks.
      不适用于 hard sequence labeling tasks。（个人觉得是因为使用的先验知识过于 general，但是 hard task 需要一些 specific 的表征。

基于此，作者提出 P-Tuning v2，作为 P-Tuning v1 的增强版，同时也是「Deep Prompt Tunning」（Li and Liang, 2021）的一个优化版本。
这种方法最主要的不同点在于：在预训练模型的每一层上使用「continuous prompt」，而不仅仅是只在输入层加入「continuous prompt」。
该方法适用于多数 NLU 任务，并且取得了和 fine-tuning 效果相当的表现，但是需要训练的参数量要小得多（0.1%-3%）。

Deep Prompt Tuning

P-Tuning v1 中，「continuous prompt」是直接被插入到「embedding层」中的，这带来了两个弊端：

  • 因为长度限制（拼接后最大长度 512），导致训练参数较少。
  • 输入层 embedding 不能直接作用于模型预测：从第一层到最后一层输出，中间有很多非线性激活函数。

    基于此，作者借鉴了「Deep Prompt Tunning」的思路，在每一层中都加入了「prefix tokens」。
    同时不同层的「continuous token」虽然位置相同，但是相互独立（而不是由之前的transformer层计算）。
    这样一来，引入了更多可训练参数，并且让 prompt 能更加直接地影响输出。

Optimization and Implementation

Reparameterization

先前的工作中，使用了一个重参数化的 encoder，来转换可训练的 embedding。
作者指出：对于一些数据集（如RTE和CoNLL04），MLP的重新参数化带来了比嵌入更稳定的改善；对于其他的数据集，重新参数化可能没有显示出任何效果（如BoolQ），有时甚至更糟（如CoNLL12）。所以本文中，该方法被移除，直接对「continuous token」进行训练。

Prompt Length

实验表明，简单分类任务的prompt一般小于 20，而难序列标注任务的长度一般为 100 左右。

Multi-task Learning

实验表明，在一些困难的序列任务中，多任务学习可以给 P-tuning v2 带来增益效果。

Classification Head

先前的工作都是使用「language modeling head」，即 Verbalizer，来预测 token。
但是作者认为在全量数据有监督场景下，没有必要使用这种 MLM 的方式进行预测。同时这种预测也不适用于序列标注问题或者其他 no-actual-meaning labels 的任务。
本文中作者直接使用了 [cls]/「token representation」+MLP 进行预测。
