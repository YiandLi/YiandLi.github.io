Direct Preference Optimization

DPO的方法跳过了奖励建模步骤，直接使用偏好数据优化语言模型，同时结合了一个动态的、每个样本的重要性权重，用于防止在朴素概率比（naive probability ratio）目标下会发生的模型退化。

PPO 需要先使用「偏好损失」来训练奖励模型，，然后冻结奖励模型去微调目标模型。

DPO 则是使用变量的变化把「偏好损失」直接定义为策略函数，也就不需要额外的奖励模型。

LDPO(πθ;πref)=−E(x,yw,yl)∼D[log⁡σ(βlog⁡πθ(yw∣x)πref(yw∣x)−βlog⁡πθ(yl∣x)πref(yl∣x))]\mathcal{L}_{\mathrm{DPO}}\left(\pi_\theta ; \pi_{\mathrm{ref}}\right)=-\mathbb{E}_{\left(x, y_w, y_l\right) \sim \mathcal{D}}\left[\log \sigma\left(\beta \log \dfrac{\pi_\theta\left(y_w \mid x\right)}{\pi_{\mathrm{ref}}\left(y_w \mid x\right)}-\beta \log \dfrac{\pi_\theta\left(y_l \mid x\right)}{\pi_{\mathrm{ref}}\left(y_l \mid x\right)}\right)\right]LDPO​(πθ​;πref​)=−E(x,yw​,yl​)∼D​[logσ(βlogπref​(yw​∣x)πθ​(yw​∣x)​−βlogπref​(yl​∣x)πθ​(yl​∣x)​)]

可以看到，我们同时跳过了显示的奖励建模步骤，并避免了强化学习的过程。

目标是：因为 reference model 不变，让 ywy_wyw​ 的「新模型生成概率」相对「原始模型的生成概率」变大，让 yly_lyl​ 的相对概率变小。

也就是让模型更倾向于生成 ywy_wyw​。

what matters is that the chosen rewards are higher than the rejected rewards (the model assigns higher reward to the chosen responses than the rejected ones).

Loss 求导分析

∇θLDPO(πθ;πref)=−βE(x,yw,yl)∼D[σ(r^θ(x,yl)−r^θ(x,yw))⏟higher weight when reward estimate is wrong [∇θlog⁡π(yw∣x)⏟increase likelihood of yw−∇θlog⁡π(yl∣x)⏟decrease likelihood of yl]]\nabla_\theta \mathcal{L}_{\mathrm{DPO}}\left(\pi_\theta ; \pi_{\mathrm{ref}}\right)= -\beta \mathbb{E}_{\left(x, y_w, y_l\right) \sim \mathcal{D}}[\underbrace{\sigma\left(\hat{r}_\theta\left(x, y_l\right)-\hat{r}_\theta\left(x, y_w\right)\right)}_{\text {higher weight when reward estimate is wrong }}[\underbrace{\nabla_\theta \log \pi\left(y_w \mid x\right)}_{\text {increase likelihood of } y_w}-\underbrace{\nabla_\theta \log \pi\left(y_l \mid x\right)}_{\text {decrease likelihood of } y_l}]]∇θ​LDPO​(πθ​;πref​)=−βE(x,yw​,yl​)∼D​[higher weight when reward estimate is wrong σ(r^θ​(x,yl​)−r^θ​(x,yw​))​​[increase likelihood of yw​∇θ​logπ(yw​∣x)​​−decrease likelihood of yl​∇θ​logπ(yl​∣x)​​]]

损失函数的梯度会增加生成更优回答 ywy_wyw​ 的概率，降低非最优回答 yly_lyl​ 的概率。

而权重中 r^θ(x,y)=βlog⁡πθ(y∣x)πref(y∣x)\hat{r}_\theta(x, y)=\beta \log \frac{\pi_\theta(y \mid x)}{\pi_{r e f}(y \mid x)}r^θ​(x,y)=βlogπref​(y∣x)πθ​(y∣x)​ ，是由语言模型 πθ\pi_\thetaπθ​ 和参考模型 πref\pi_{r e f}πref​ 隐式定义的奖励 ，可以表示是不是首选回答。从而表示了隐式奖励模型对回答进行错误排序的程度 ，体现出 KL 约束的强度。在所有可能的输出结果排序中，如果 yly_lyl​ 排名在 ywy_wyw​ 前面，则这个差值为 + ，而且排名差距越大，差值越大。

训练过程

  • 对每个 prompt 进行采样：y1,y2∼πref(⋅∣x)y_1, y_2 \sim \pi_{r e f}(\cdot \mid x)y1​,y2​∼πref​(⋅∣x) ， 基于人类偏好标注并构建离线的偏好数据集 D={x(i),yw(i),yl(i)}i=1N\mathcal{D}=\{x^{(i)}, y_w^{(i)}, y_l^{(i)}\}_{i=1}^ND={x(i),yw(i)​,yl(i)​}i=1N​ 。
  • 开始训练时，reference model和policy model都是同一个模型，在训练过程中reference model不会更新权重。
    直接通过 LDPO(πθ;πref)\mathcal{L}_{\mathrm{DPO}}\left(\pi_\theta ; \pi_{\mathrm{ref}}\right)LDPO​(πθ​;πref​) 进行优化 。
    当然同样可以考虑通过最大化最优问答对 (x,yw)\left(x, y_w\right)(x,yw​) 来得到初始模型，这个过程有助于减小真实参考分布（不可知）和 DPO 实际使用的  之间的分布偏移。

讨论

  • 在 benchmark 上 DPO 确实会好，但因为它没有 reward model，理论上是个behavior clone，它没有OOD能力的，可以理解成蒸馏reference policy。
  • 在比较容易用大量的正负样本数据表达希望模型调整的方向，RLHF DPO 会比较容易。
  • DPO 使用一个最大似然目标学习策略，能够同时跳过奖励评估和强化学习的过程。
  • 针对一个 prompt，PPO 只需要生成一个 response，DPO 需要生成两个 response 。

通过不同的方式(ChatGPT、人类专家等)生成很多个answer，让RM去打分，然后选出好的去Fine-tune大模型。

此外，让大模型也算一下这些answers的似然概率当做评分，让这个评分和RM的打分尽可能对齐（通过Rank loss的监督学习目标）。

  3. Token-wise有考虑累计回报奖励的潜力；
  4. 相比于直接用样本进行最大似然，让模型自己去生成可以允许更多样化的输出（RRHF和RAFT也有这个优势）。

显存受限

在现代GPU中，计算速度已经远超过了显存访问速度，transformer中的大部分计算操作的瓶颈是显存访问。
对于显存受限的操作，IO感知是非常重要的，因为显存读写占用了大部分的运行时间。

为什么呢？来具体分析下。

  • 计算带宽（math bandwidth） 指的是处理器每秒钟可以执行的数学计算次数，单位通常是OPS(即operations/second)。如果用浮点数进行计算，单位是FLOPS。

  • 内存带宽 （memory bandwidth） 指的是处理器每秒钟从内存中读取的数据量，单位是bytes/second。

在计算科学中，给定硬件，找出一个操作的性能瓶颈是非常重要的。
一个操作的性能瓶颈有两类：计算受限（math-bound）和内存受限（memory-bound）。

从数学上分析，浮点数计算次数为 NopN_{o p}Nop​ ，内存访问量为 $N_{b y t e } $ 计算带宽为 $ B W_{m a t h} $ ，内存带宽为 $ B W_{m e m} $ 。

访问内存花费的时间为 TmemT_{m e m}Tmem​ ，计算花费的时间为 TmathT_{m a t h}Tmath​ 。

Tmem =Nbyte BWmem Tmath =NopBWmath \begin{aligned} T_{\text {mem }} & =\frac{N_{\text {byte }}}{B W_{\text {mem }}} \\ T_{\text {math }} & =\frac{N_{o p}}{B W_{\text {math }}} \end{aligned} Tmem ​Tmath ​​=BWmem ​Nbyte ​​=BWmath ​Nop​​​

由于可以"边计算上一个，边读/写下一个"，访问内存和计算的时间可以重叠，此时总的运行时间为 max⁡(Tmem ,Tmath )\max \left(T_{\text {mem }}, T_{\text {math }}\right)max(Tmem ​,Tmath ​) 。

当 Tmath>TmemT_{math}>T_{m e m}Tmath​>Tmem​ 时，硬件性能受限于计算带宽，为计算受限 (math-bound)。在数学上，下面的不等式成立：

Tmath >Tmem →NopNbyte >BWmath BWmem T_{\text {math }}>T_{\text {mem }} \quad \rightarrow \quad \frac{N_{o p}}{N_{\text {byte }}}>\frac{B W_{\text {math }}}{B W_{\text {mem }}} Tmath ​>Tmem ​→Nbyte ​Nop​​>BWmem ​BWmath ​​

反之，当 $ T_{math}<T_{m e m}$ 时，硬件性能受限于内存带宽，为内存受限 (memory-bound)。在数学上，下面的不等式成立：

Tmath <Tmem →Nop Nbyte <BWmath BWmem T_{\text {math }}<T_{\text {mem }} \quad \rightarrow \quad \frac{N_{\text {op }}}{N_{\text {byte }}}<\frac{B W_{\text {math }}}{B W_{\text {mem }}} Tmath ​<Tmem ​→Nbyte ​Nop ​​<BWmem ​BWmath ​​

通常将 ops/bytes =NopNbyte =\frac{N_{o p}}{N_{\text {byte }}}=Nbyte ​Nop​​ 定义为算术强度（arithmetic intensity）。给定硬件的情况下，对于一个模型或算法，若算术强度 >BWmath BWmem >\frac{B W_{\text {math }}}{B W_{\text {mem }}}>BWmem ​BWmath ​​ ，则为计算受限；反之为内存受限。
假如用float16进行计算，A100-40GB SXM的 BWmath BWmem =312×10121555×109≈201\frac{B W_{\text {math }}}{B W_{\text {mem }}}=\frac{312 \times 10^{12}}{1555 \times 10^9} \approx 201BWmem ​BWmath ​​=1555×109312×1012​≈201 flops /bytes 。若一个操作的算术强度 ops/types >201 ，此时的性能受限于计算带宽；反之，性能受限于内存带宽。

Attention 中：矩阵乘是计算受限，而 softmax ，dropout，mask，reduction操作 等是内存受限。

所以：内存受限 --> 减少HBM读写次数 --> kernel融合 --> 满足SRAM的内存大小 --> 分块计算

  • 显存分析命令 ：

    不要使用 nvidia-smi ，因为 PyTorch是有缓存区的设置的，意思就是一个 Tensor 就算被释放了，进程也不会把空闲出来的显存还给 GPU，而是等待下一个 Tensor 来填入这一片被释放的空间 。所以不能准确地给出某一个时间点具体的 Tensor 占用的显存，而是显示的已经分配到的显存缓冲区和 Torch 在创建cuda进程时所需开销的和，也就是 reserved_memory 和 torch context 显存之和。

    torch.cuda.memory_allocated() ：精准地反馈当前进程中 torch.Tensor 所占用的 GPU 显存 (注意是只包括torch.Tensor)

    torch.cuda.max_memory_allocated() ：告诉我们到调用函数为止所达到的最大的显存占用字节数

  • Cuda Context

    https://discuss.pytorch.org/t/how-do-i-create-torch-tensor-without-any-wasted-storage-space-baggage/131134

    PyTorch will create the CUDA context in the very first CUDA operation, which can use ~600-1000MB of GPU memory depending on the CUDA version as well as the used device. PyTorch itself will allocate the needed memory and will use an internal cache mechanism.

  • 训练过程显存分析：

    模型参数、优化器状态和参数梯度是始终在占用着存储空间的，中间激活值在反向传播之后就自动被清空了。

    定义模型：模型参数 * 4 （ float32 每个参数占用 4 bytes ）

    前向传播：

      input 在输入模型前需要计算显存

  然后生成激活值显存，显存增加等于每一层模型产生的结果的显存之和，且跟batch_size成正比。

    反向传播：

      主要是为了求 loss

  后向传播会将模型的中间激活值给消耗并释放掉；然后留下梯度占用的激活值。

  在第一次执行时，会首先分配特定的空间，用于存储梯度。

  然后这部分空间基本就不变了，即便模型进行更新，也不会变化，会等待用于存储下一批梯度。

    参数更新：

      第一次执行（`.step()`）时，会为每一个参数初始化其优化器状态，对于这里的AdamW而言，每一个参数需要额外的4*2=8个字节。 

  第二次开始，不会再额外分配显存。

  由于计算机计算的特性，有一些计算操作在计算过程中是会带来额外的显存开销的。但是这种开销在torch.memory_allocated中是不能被察觉的。

  比如在AdamW在进行某一层的更新的时候，会带来2倍该层参数量大小的临时额外开销。这个在max_memory_allocated中可以看到。

float16 比float32小一半，显存占用可以降低一半；减少网络通信的开销；硬件针对float16有优化，速度更快；

但是，float16最大的问题是"下溢"。模型的更新通常是 gradient×lr\text{gradient}×\text{lr}gradient×lr，随着模型的训练，这个值往往会很小，可能会超出float16表示的精度。

导致模型权重都不再更新，模型难以收敛。

所以使用混合精度训练，模型权重、梯度使用float16，优化器参数为float32。此外，优化器还需要保存一份float32版本的权重。

混合精度的具体过程如下：

  1. 使用float16权重进行前向传播；
  2. 反向传播得到float16的梯度；
  3. 通过优化器计算出float32精度的权重更新量；
  4. 更新float32权重；
  5. 将float32权重转换为float16；

推理时间估计

N 为 模型参数量
prefill 阶段，s 比较大，计算时间 >> 搬运时间，所以为计算密集型。
decode 阶段，s = 1，计算时间 << 搬运时间，所以为传输密集型。

Latency：延迟 ，即从输入到输出最后一个 token 的延迟。
Latency = (TTFT) + (TPOT) * (the number of tokens to be generated). Latency 可以转换为 Tokens Per Second (TPS)：TPS = (the number of tokens to be generated) / Latency。

Throughput：吞吐量 ，即每秒针处理的请求数 / 生成的token数量。
Prefill 阶段在比较小 Batch Size 下可以获得比较大的计算强度，相应的吞吐也很高；而 Decoding 阶段需要比较大的 Batch Size 才能获得相对高的计算强度及吞吐

关注点：
在线流式应用： 对 TTFT、TPOT、Latency 敏感，需要尽可能快的生成 token。
离线批量应用： 对 Throughput 敏感，需要在单位时间内尽可能多的生成 token。
