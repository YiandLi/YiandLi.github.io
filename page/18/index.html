7.1 分布式原语

Posted on 2022-03-23 In Coding , PyTorch
Symbols count in article: 677 Reading time ≈ 3 mins.

Pytorch - 分布式通信原语（附源码）

Torch 分布式训练依赖于 torch.distributed 模式，提供两种通信方式：

  1. point 2 point communication: send,recv
  2. collective communication: scatter, broadcast, gather, reduce, all_reduce, all_gather

API-Collective functions: https://pytorch.org/docs/stable/distributed.html#collective-functions
API-Point-to-point communication: https://pytorch.org/docs/stable/distributed.html#point-to-point-communication

broadcast

将某一个 tensor 从 source rank 广播到其他 rank 。

dist.broadcast(tensor, src = 0) ，这是一个 in-place 方法，没有返回值，参数 src 设置哪个 rank 进行 broadcast，默认为 rank 0 。

首先创建一个张量，广播后这个张量被覆盖。

Scatter

将一个 tensor 切分并且均匀分布到其他 rank 上。

torch.distributed.scatter(tensor, scatter_list=None, src=0, group=None, async_op=False)

  • tensor：除 src rank 外，其他 rank 获取并且存储 output tensor 的参数
  • scatter_list：需要进行 scatter 的 tensor list，只有 src rank 才能设置 scatter_list 的值。这个值需要和 dest rank 数量相同。
  • src：设置哪个 rank 进行 scatter，默认为 rank 0

注意上述代码中，scatter_list 类型为 list 。
一共有4个rank参与了scatter计算，
计算之前：scatter list为 [0,0], [1,1], [2,2], [3,3];
计算之后：rank 按顺序被分配 scatter list 的每一个 tensor，rank0 为 [0,0]，rank1 为 [1, 1]，rank2 为 [2, 2]，rank3 为 [3, 3];

gather

和 scatter 相反，将多个元素汇总到同一个rank上。

torch.distributed.gather(tensor, gather_list=None, dst=0, group=None, async_op=False)

  • tensor：所有rank的input tensor
  • gather_list：dst rank 的 output 结果；只有 dst rank 需要设置；长度需要和 src node 数量一致。
  • dst：汇总数据的 node id

reduce

可以看成是对 gather 的结果求和 。

torch.distributed.reduce(tensor, dst, op=<ReduceOp.SUM: 0>, group=None, async_op=False)

  • tensor：需要进行reduce计算的数据，对于dst rank来说，tensor为最终reduce的结果
  • dist：接收答案的 rank id
  • op：聚合算子 ，包含 SUM, PRODUCT, MIN, MAX, BAND, BOR, BXOR 。

注意，这里有个副作用，就是其他 source rank 的 tensor 也会被修改。

all-gather

torch.distributed.all_gather(tensor_list, tensor, group=None, async_op=False) ，每个 rank 都要指定 gather_list 。

all-reduce

torch.distributed.all_reduce(tensor, op=<ReduceOp.SUM: 0>, group=None, async_op=False)

badcase

Posted on 2022-03-14 In NLP
Symbols count in article: 1.5k Reading time ≈ 7 mins.

source: 心法利器[37] | bad case治疗术：认知篇

现状篇

  • 使用完整的评测集来进行检验，而不是单纯的一两个样本。
  • 需要一个具有统计意义上的数据集：
    • 数量上能支撑特定指标的计算足够稳定。
    • 抽样情况和实际使用的场景尽可能一致。
    • 质量可靠，足够支撑当前方案的数据集。
  • 指标
  • 结论

分析篇

准确率分错的case，看漏召回的case

粗看

笼统地大体上去看，错误的样本都有什么特点，带有那些特征，例如长度、句式等是否有什么特点，又或者是问句等等。
这个只要通篇看下来，就会有比较明显的特点了，一下子能总结出比较突出的问题。
快速看快速找到比较大的问题，优点就是快，但问题是你要能粗看的出来才行，所以缺点就是细节问题比较难发现，而且有些问题是比较深层次的，例如同样是问句，但是为什么问句就会出错

追溯法

对一个case，重现整个训练和预测过程的方法。给定一个bad case，准备好日志，分析预测的每个阶段的分析结果，查看是否符合预期，不符合预期的，大概率这个部分有bug、词典词汇等什么明显的问题了。

比如，很多问句会被误召回，这时候我们可以查查训练集中带有问句关键词的正负样本的比例是不是有问题，例如正样本远多于负样本，此时模型很可能就会认为只要带有疑问词的就是正类，我们需要加一些带疑问词的负样本进去。

解决法

例如短句预测不正确，而且一般都比较模糊，所以我们考虑的是过滤所有短句，然后再通过白名单的方式补召回，那我们就可以把query归类为短句。

问题标记

case是需要我们逐个分析的，后续我们需要对重点问题进行归纳总结，所以case分析往往是需要我们做一些关键的记录。

query 原因分析
关闭 短句
今天天气怎么样 疑问词，关键词拒绝
你为什么 疑问词，模糊

标记的时候注意词汇和问题的统一，方便我们后续统计各个问题的占比，值得注意的是，一个问题是可能有多个解决方法的，我们记得要标记上，统计分析的时候都是要关注的。

问题统计和归纳

有了上面的标记，我们能很轻易地看拿到各个问题的分布情况，甚至可以总结出来。

解决篇

样本的误导

对于一些数据内很浅层直白的信息，模型也会预测的非常直白，例如正类里面有大量含有“怎么”的样本，而负类里面没有，则模型很有可能会把所有带有“怎么”的句子都预测为正类。这个时候就需要负采样了。

阈值的确定和权衡

阈值本质是个准入条件，是和模型一起出现的标配，毕竟模型大部分时候只提供意见，要不要采纳还是要通过阈值（参考sigmoid，一般阀值为0.5）。
阀值变小，召回率变高，精确率下降，我们需要确定一个最佳阈值，对于搜索和对话场景，更多的是需要我们保准确，一般是0.85+，召回率另说。

softmax 呢？相当于双重条件？max+大于阀值

前后处理和预处理

不同的场景和数据的预处理方式会很不同，哪怕是同样的NLP问题，到了细分场景也会有很多不同的操作，共同的可能是对标点的处理，而不同的例如客服场景要把“你好”之类的删除，语音助手要把“小爱同学”、“小布”之类的称呼删除等，重点不在于做了什么，而是要把一些对预测有影响干扰项、不利于预测的因素给去掉，至于哪些是干扰项，请回去看bad case，这些错误样本会告诉你。

需要把一些肯定确定能快速预测出来的东西给过滤掉，从而提升性能也降低模型的负担，最简单的例如黑白名单，复杂的可以有一些针对业务的规则，例如超短句或者超长句直接拒绝等等，能用规则的尽量用规则，能不用模型的尽量不用模型，模型的不可控性众所周知，我们知道修复bad case的困难性的情况下，没必要太往依赖模型了。

后处理是指在模型预测以后的一些调整，最直接能想到的就是阈值过滤，但不仅是这些，有的时候需要结合模型的预测打分进行调整，例如关键词抽取领域，需要对预测的重要性分数进行加减分，如句首词和句尾词要提权，对于百科的一些问题，语义相似度的定义可能会和实际应用不同，于是我们也要进行一些过滤，例如“五一节的由来”和“六一节的由来”从语义相似度上来说确实很接近，但是我们不能然你给他们相似，因为“五一节”和“六一节”很不同，所以我们通过“实体约束”来过滤掉他们，当然背后就需要一批实体词典了。

转置卷积层（Transpose Convolution Layer），在一些文献中也称之为反卷积(De-convolution)和部分跨越卷积(Fractionally-strided Convolution) 。

从实现上来说，卷积本身是不可逆的，这里的反卷积仅仅是正常卷积 尺寸 上的逆向，内容不一定是逆向关系。所以除了提高分辨率，还可以进行风格迁移，理论上只要是 domain 2 domain 的任务，都可以进行。

一个卷积操作表示一个 多对一(many-to-one) 的映射关系。
而转置卷积层我们表示一个 一对多(one-to-many) 的映射关系。

从普通卷积讲起

假设我们有一个 4×44\times 44×4 的图片，同时也有一个 $3\times 3 $ 的卷积核 （不考虑通道数），步长为 1 ，则最后会得到一个 2×22 \times 22×2 的输出。

如果我们把每一步的卷积核对应原始图片位置进行 flatten ， 则会得到 1×161 \times 161×16 的矩阵：
比如在第一步骤中：

X 为第一步骤没被卷积计算的部分。

每一时间步骤的「卷积相对矩阵」为一行，将所有时间步骤的「卷积相对矩阵」全部 flatten 并且拼接在一起，可以得到一个 4×164\times 164×16 的矩阵：

相应地，我们也将图片进行 flatten，得到一个 16×116 \times 116×1 的矩阵。

然后相乘，得到了 4×14\times 14×1 的矩阵，正好对应了 2×22\times 22×2 的卷积结果。


更一般地：
使用 n2×1n^2 \times 1n2×1 表示图像 ；
使用 $\text{time\ step}\ \times\ n^2 $ 的矩阵表示卷积核 ，n 为图片height / weight ；
最终输出结果为 time step×1\text{time\ step}\times 1time step×1

所以，如果可以改变 time step\text{time\ step}time step 那么，就可以改变输出的尺寸。
而同时如果反过来， output 变成了 input 2×22\times 22×2 ，「卷积相对矩阵」被转置了 16×416 \times 416×4 ，那么就可以得到原始图像 (16×4)⋅(2×2∗1)=(16×1)(16 \times 4)\cdot(2\times 2*1) = (16\times 1)(16×4)⋅(2×2∗1)=(16×1)

所以本质上，我们需要学习一个矩阵，大小为 outputsize2×input2output size ^ 2 \times input ^ 2outputsize2×input2 ，即转置矩阵。

在具体实现上，大多数情况下转置矩阵是用不到卷积矩阵的参数的，只是根据卷积矩阵推导出 一个shape,让网络能够上采样得到之前大小图片。
