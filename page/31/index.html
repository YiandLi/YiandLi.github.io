深度学习框架的优势

  1. 简化计算图的搭建
  • 计算图是一种描述函数的语言，其中节点代表输入值，边代表函数操作。
  • 计算图可以用于描述表达式，可以更加精确描述网络中的参数传播过程。

  1. 简化偏导数的计算
    计算图以模块化的方式完整表达了函数体的内部逻辑关系，所以对复杂函数求导的过程就变成了简单地从输入到输出进行遍历的过程。绝大多数深度学习框架都选择基于计算图的声明式求解。

  2. 高效运行
    移植性：可以将代码部署到GPU或CPU上

深度学习的单层神经网络 – 逻辑回归

  • 典型的深度学习的计算过程包含三个过程：
    1. 正向传播 Forward Propagating
    2. 反向传播 Backward Propagating
    3. 梯度下降 Gradient Descent
  • 损失函数：线性回归中，损失函数通常为 L(y^,y)=12(y^−y)2L(\hat{y},y)=\frac{1}{2}(\hat{y}-y)^2L(y^​,y)=21​(y^​−y)2，属于凸优化问题，但是在逻辑回归中，该损失函数的优化问题将会变成非凸的。
    所以逻辑回归中通常使用对数损失函数(Logarithmic Loss Function): L(y^,y)=−[ylogy2+(1−y)log(1−y^)]L(\hat{y},y) = -[ylog y^2 + (1-y)log(1-\hat{y})]L(y^​,y)=−[ylogy2+(1−y)log(1−y^​)]
  • 成本函数：全部训练样本的总成本：J(w,b)=1m∑L(y^i,yi)J(w,b)=\frac{1}{m}\sum{L(\hat{y}^i,y^i)}J(w,b)=m1​∑L(y^​i,yi)
  • 向量化：通过矩阵相乘来代替循环遍历的诸葛相乘以缩短训练时间。

凸函数和凸优化问题

  • 凸函数：函数表示的区域任意两点连成的直线上的任意一点都落在该函数表示的区域内部
  • 凸优化问题：定义在凸集中的凸函数最优化的问题。局部最优解同时也是全局最优解

激活函数

作用

  • 中间层的激活函数：数据的分布绝大多数是非线性的，而一般神经网络的计算是线性的，引入激活函数，是在神经网络中引入非线性，强化网络的学习能力。所以激活函数的最大特点就是非线性。
  • 输出层激活函数：
    线性回归中，不需要激活函数。
    多分类中，直接使用softmax求出logistic值/置信度，后加最小化熵损失。
    二分类中，需要使用sigmoid激活函数，压到[0,1]之间得到logistic值/，后加ce求损失。

不同的激活函数

  • Sigmoid激活函数：σ(x)=11+e−x\sigma(x) = \frac{1}{1+e^{-x}}σ(x)=1+e−x1​
    • 取值范围：(0, 1)
    • 缺点：计算量大，反向传播求误差梯度时，求导涉及除法，
    • Sigmoid导数取值范围是[0, 0.25]，由于神经网络反向传播时的“链式反应”，很容易就会出现梯度消失的情况。例如对于一个10层的网络， 第10层的误差相对第一层卷积的参数的梯度将是一个非常小的值，这就是所谓的 “梯度消失”。
    • Sigmoid的输出不是0均值（即zero-centered）；这会导致后一层的神经元将得到上一层输出的非0均值的信号作为输入，随着网络的加深，会改变数据的原始分布。
  • tanh激活函数：tanh(x)=ex−e−xex+e−xtanh(x) = \frac{e^{x}-e^{-x}}{e^{x}+e^{-x}}tanh(x)=ex+e−xex−e−x​
    • tanh的输出范围时(-1, 1)，解决了Sigmoid函数的不是zero-centered输出问题
    • tanh导数范围在(0, 1)之间，相比sigmoid的(0, 0.25)，梯度消失（gradient vanishing）问题会得到缓解，但仍然还会存在。
  • ReLu激活函数：relu=max(0,x)relu=max(0, x)relu=max(0,x)

    • 对ReLU求导，在输入值分别为正和为负的情况下，导数是不同的
    • Relu的输入值为负的时候，输出始终为0，其一阶导数也始终为0，这样会导致神经元不能更新参数，也就是神经元不学习了，这种现象叫做“Dead Neuron”。

性质

  • f(x)≈xf(x)≈xf(x)≈x ：
    当激活函数满足这个性质的时候，如果参数的初始化是random的很小的值，那么神经网络的训练将会很高效；如果不满足这个性质，那么就需要很用心的去设置初始值
  • 输出值的范围：
    当激活函数输出值是有限的时候，基于梯度的优化方法会更加稳定，因为特征的表示，当激活函数的输出是无限的时候，模型训练会更加的高效，不过在这种情况很小，一般需要一个更小的学习率

batch-size 和 epoch

  • batch_size：每批数据量的大小，一次训练batchsize个样本，计算它们的平均损失函数值，来更新参数。
  • epoch：1个epoch指用训练集中的全部样本训练一次。

batch_size的大小决定了是否可以表示全体样本的特征，梯度越准，手链速度越快。
batch_size过大：计算量太大，内存占用太多，更新速度过慢。并且缺乏随机性，容易使梯度始终向单一方向下降，陷入局部最优。
batch_size过小：每次训练的速度快，但是因为样本随机性较高，收敛速度很慢，产生局部震荡，有限迭代次数内很可能无法收敛。

K近邻算法

描述

  1. 首先设定参数为k
  2. 将新数据的每个特征与样本数据中的每个特征进行比较，选出样本数据中与新数据差异最小(这里可以用距离表示)的k个实例。
  3. 然后，用这k个实例中出现次数最多的分类来作为新数据的分类。

归一化特征值 Standardization

  • 问题：如果我们选用的距离度量为欧式距离，如果数据预先没有经过归一化，那么那些绝对值大的特征值在欧式距离计算的时候起了决定性作用。
  • 描述：
    1. 梯度下降法求解的模型一般需要归一化。
    2. 一般关注特征值的具体值的模型需要归一化处理，而一般关注特征值的分布情况的则无需归一化。
  • 公式：
    1. Min-Max/Linear normalization 线性函数归一化： newValue = (oldValue - min) / (max - min) ，最终结果的值域为 [0, 1]
    2. Z-score normalization 零均值归一化： newValue = (oldValue - 均值) / 标准差 , 最终结果值域为 [-1, 1]
    3. Sum normalization 特征归一化：newValue = oldValue / featureSum, featureSum是指所有实例的该特征值的和， 最终结果值域为 [0, 1]

多表操作的结果是一个新的临时表，一般是用于 FROM 后面，然后进行新的 SELECT。

表文件的关系

  • [Coding, MySQL]通过表文件的数据关系来描述现实生活中的隶属关系。

  • 隶属关系角色划分：

    1. 一方：拥有资源的一方
    2. 多方：依附于一方的资源

  • 三种关系

    • 一对一

      • e.g. 用户和用户详情
        常用于单表拆分，将一个列数很多的用户信息表，拆分称为一个只包含基础信息的小表（列数小）和一个包含其他信息的大表（列数大），来提升操作效率。
      • 实现：
        在任意的一个表中加一个 UNIQUE 外键，关联另一个表的主键。
        UNIQUE 保证了一对一的关系。

    • 多对多

      • e.g. 学生-课程：一个学生可以选修多门课，一门课可以被多个学生选择。
      • 实现：建立第三个中间表（选课表），中间表的一个外键关联学生的主键，一个外键关联课程的主键。

    • 一对多 / 多对一

      • e.g. 部门-员工：一个部门对应多个员工，一个员工只对应一个部门。
      • 实现：多方表（员工）建立外键，指向一方表（部门）的主键。

表文件字段的分类

  • 主键字段：
    1. 存放主键编号，每一个表都应该有一个主键字段。
    2. 用来区分表中的唯一的一个元组/对象。
    3. 不为null，同一表中不可重复。
  • 非主键字段：描述主键编号。
  • 外键字段：
    1. 只存在多方表，描述多方数据与一方数据的依赖关系。
    2. 外键字段的值应该来源于一方表中的主键值。

多表查询

  • 本质：将多张表数据合并到同一个临时表中，然后使用六个查询命令，根据需求得到需要的数据。
  • 关键：如何将多张表中的数据合并到同一张表中。
  • 多张表数据合并方案：
    1. 连接查询 合并方案
      a. 首先确保两张表之间存在了隶属关系。
      b. 将两张表的内容水平方向拼接。
      c. 最终得到拥有完整隶属关系的新数据行。
    2. 联合查询 合并方案
      a. 不需要两张表存在隶属关系。
      b. 两张表的数据行沿着垂直方向堆砌。


连接查询合并方案

例子：查询各个部门名称及其部门下的职员名称。

  • 数据来源：Dept.dname 和 Emp.ename
  • 表文件的关系：Emp隶属于Dept
  • 命令：join
  • 运算过程：
  1. JOIN 将两个表水平拼接生成一张新的临时表，字段由两个表相加而成，新的字段名格式为表名.字段名，这样解决了重复字段名的问题。
  2. 临时表行数为原表行数做乘法，是一个笛卡尔积。
  3. 这时候行数很多，而且每一行是直接拼接的，很多数据根本不合法（脏数据）。这时候我们就需要进行过滤操作了 。
    Join 过滤操作会删除不合法的行，即主客键数据冲突的信息（脏数据），然后放置在一个新的临时表中，接受后续操作。

连接查询过滤方案：将合法数据行与脏数据行进行剥离的形为。

  • 命令格式： FROM 一方表 [INNER/LEFT/RIGHT] JOIN 多方表 ON 定位合法数据行条件
  • 合法数据行与脏数据行区分原则：
    1. 表文件存在明显外键字段：ON 一方表.主键字段 = 多方表.外键字段
    2. 表文件没有明显外键字段：ON 根据实际隶属关系进行判断
  • 分类：内连接过滤方案，外连接过滤方案，自连接过滤方案

内连接查询

不指定字段

使用 natural join ，不需要用户声明需要匹配的特征名称，会自动将两个表中具有相同名称的列进行匹配。

指定字段方式

查询两个表交集部分的数据

  • 命令格式：

    • 隐式：SELECT 字段列表 FROM 表1, 表2 WHERE 条件 ...;
    • 显式：SELECT 字段列表 FROM 表1 [INNER] JOIN 表2 ON 连接条件 ...;，INNER 关键字可以省略。

  • 例子：

  • 顺序：

    1. FROM Dept JOIN Emp
    2. ON Dept.Deptno = Emp.Deptno
    3. WHERE Dept.Deptno=20
    4. GROUP BY Emp.job 生成两个临时表
    5. SELECT Dept.Dname, Emp.job, COUNT(*) 对每一个临时表进行操作

  • 简化：别名
    在SELECT,ON,WHERE中都需要写下表名.字段名，比较繁琐，这时我们可以在FROM处给表起别名。
    但是起了别名以后的操作，就不能通过表名.字段名来指定别名了，只能通过别名.字段名的方式指定。

    • 右外连接：查询右表所有数据，和两张表的交集部分。

Null 值现象

首先空值 '' 也是一个值，和普通值一样的，和 null 没关系。
在 MySQL 中，NULL 值与任何其它值的比较（即使是 NULL）永远返回 NULL，即 NULL = NULL 返回 NULL ，而我们筛选是需要返回 True / False 。

left join 时，如果多方表中没有和一方表中匹配的选项，那么一方表中的该 tuple 也会被输出，但是多方字段就是 null 了。

可以看到 |2 |null|null|null| 和 |5 |null|null|null| ，A.NAME=Null，并未关联到B表NAME为NULL的部分 B: |3 |null| ，所以 一方表中的 null，不会匹配多方表中的任何结果，即使多方表中也是 null 。

解决方法：

  1. 让 Null 也去匹配多方表中的 Null 或者 ''
    使用 COALESCE 关键字，COALESCE(A.NAME,'')，如果是 Null 则返回空值 。
    因为左右都将 Null 转换为了 ''，所以一方表中的 Null 可以匹配到多方表中的 Null 和 '' ，并且返回表还会显示原始的元素值。

COALESCE (expression_1, expression_2, ...,expression_n) 依次参考各参数表达式，遇到非 null 值即停止并返回该值。如果所有的表达式都是空值，最终将返回一个空值。

子查询

  • 定义：在查询时无法从当前临时表得到需要的数据，此时mysql服务器允许开发人员通过完整查询语句从别的表中得到需要的数据，从而保证查询正常执行。
    这个起到帮助作用的完整查询语句相当于外援，在技术上被称之为子查询或者嵌套查询。

  • 适用条件：根据某些限制特征来得到某一目标群体的其他特征

  • 用法：使用 小括号 放置查询命令，令其返回一个值。可以看成调用了一次简单的函数。

  • 命令：SELECT * FROM t1 WHERE column1=(SELECT column1 FROM t2)

  • 根据查询结果，即 小括号() 内部查询命令返回值的不同，分为四种子查询：

    • 标量子查询：返回结果为一个标量
    • 列子查询：返回结果为一列
    • 行子查询：返回结果为一行
    • 表子查询：返回结果为一个临时表

  • 可用范围：

    • SELECT 命令：SELECT ename,sal,(SELECT AVG(sal) FROM emp) AS 'avg'
    • FROM 命令：把子查询的结果(内存里的一张表)当作一张临时表，然后交给外部继续对它进行处理。
    • WHERE 命令：把内层查询的结果当作外层查询的条件。
    • HAVING 命令：通过子查询来得到判断标准。HAVING avg(sal)>(SELECT avg(sal) FROM emp)
    • EXISTS 命令：对外层表进行循环，再对内表进行内层查询。
      和 in() 差不多，但是它们还是有区别的。主要是看两个张表大小差的程度。若子查询表大则用exists（内层索引），子查询表小则用in（外层索引）。

  • 运算符号：

  1. in子查询：内层查询语句仅返回一个数据列，这个数据列的值将供外层查询语句进行比较。
  2. any子查询：只要满足内层子查询中的任意一个比较条件，就返回一个结果作为外层查询条件。
  3. all子查询：内层子查询返回的结果需同时满足所有内层查询条件。
  4. 比较运算符子查询：子查询中可以使用的比较运算符如 >、>=、<=、<、=、 <>
  5. exists子查询：把外层的查询结果（支持多行多列），拿到内层，看内层是否成立，简单来说后面的返回true,外层（也就是前面的语句）才会执行，否则不执行。

关系型数据库

关系型数据库(rational database) 中的关系是 relation，定义为：

给定 nnn 个集合 S1,S2,S3,...,SnS1, S2, S3, ..., SnS1,S2,S3,...,Sn 。同时 RRR 是一个 nnn 元数组 (n-tuples)，它的第一个元素取自集合 S1S1S1 ，第二个元素取自集合 S2S2S2 ，以此类推。我们将 RRR 称之为基于该 n 个集合的一个 Relation， SjSjSj 为 RRR 的第 jjj 个域 (Domain)。

这里集合可以看成是 column 属性集合，由其元素组成的元组 RRR 可以看成每一行，即每一个实例都是一个 relation 。

所以在定义每个表时候，会先定义 “值域”（即上面所说的集合S1, S2, S3, …, Sn），然后再定义 “表”，即使用哪些集合。这种建模思想能 促进重用，例如 “性别”这个值域也可用在“教师”这个表里。这点在建立大型信息系统是极有价值的。

关系型数据库的关系模式就是二维表模型。

对象型数据库，比如 MongoDB，是 用面向对象的思想去设计数据存储模型。

非关系型数据库 NoSQL

参考博客：Sql Or NoSql，看完这一篇你就懂了

NoSQL 是随着数据越来越多，SQL的优化已经不能满足数据膨胀式的增长，而且SQL在分布式方面表现不是特别好，NoSQL就是在这种情况下产生的。

Not Only SQL，泛指非关系型数据库，是对关系型数据库的一种补充，不是替换。
NoSql与关系型数据库并不是对立关系，二者各有优劣，在合适的场景下选择合适的存储引擎才是正确的做法。

数据以键值对（kv）来存储，没有固定的(二维)结构，这样就不用局限于固定的结构。用于存储那些类型不固定的，也没有什么规律的数据。

在查询时，根据 key 就可以取出对应的 value。

优点/为什么还需要 NoSQL：

  1. 非关系型数据库是基于键值对的，数据处理过程不需要经过SQL层的解析，效率相对很高。
  2. 传统的关系型数据库很难实现水平扩展，扩容成本很大。
    而对于非关系型数据库是基于键值对，数据之间没有耦合性，能够非常容易水平扩展。

文档数据库（MongoDB）

这里说的“文档”其实是一个数据记录，这个记录能够对包含的数据类型和 内容进行 “自我描述”。
文档数据库可以包含非常复杂的数据结构，比如嵌套对象并且不需要使用特定的数据模式，每个文档可以具有 完全不同的结构。

DML (Data Manipulation Language) 数据操作语言
数据操纵分成 数据查询 和 数据更新 两类。数据更新又分成 插入、删除、和修改 三种操作。
mysql一共有7个查询命令：FROM，SELECT，WHERE，GROUP BY，HAVING，ORDER BY，LIMIT

查询命令和临时表

  1. 每一个查询命令在执行时，都是在操作上一个命令生成的临时表。
  2. 每一个命令执行完毕后，Mysql服务器自动在内存中销毁上一个查询命令生成的临时表。所以，用户只能看到最后一个查询命令生成的临时表。
  3. 只有FROM命令不需要操作临时表。
  4. 只有GROUP BY命令在执行完后有机会生成多个临时表。
  5. 只有HAVING命令在执行完毕后，不会生成新的临时表。负责将GROUP BY命令生成的临时表中不满足的临时表从内存中删除。

FROM命令

  1. 永远是第一个执行的查询命令，负责将硬盘上的表文件加载到内存中生成临时表供后续查询命令使用。所以，执行查询命令时，是不会修改文件原来的数据的。
  2. 临时表的名称与硬盘上的表文件名称保持一致。
