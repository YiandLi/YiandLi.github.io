LoRA

优点：

  1. 一个中心模型服务多个下游任务，节省参数存储量
  2. 推理阶段不引入额外计算量，推理延迟
  3. 和其他高效微调方法正交，可以组合

缺点：

  1. 参与训练的模型参数量不多，也就百万到千万级别的参数量，所以效果比全量微调差很多。
    如果有足够计算资源以及有10k以上数据，建议全参数微调。
  2. 训练时间更久

所以到底为什么能加快训练？

和原本全参数预训练相比，前向传播和后向传播在显存中没有任何节省，只能节省 优化器状态部分 。被冻结住的参数不用更新，自然也就不需要相应的Adam一阶和二阶动量。

因为被冻结住的主干模型参数不用存储相应的优化器状态，显存占用降低，可以用更多显存装输入数据和hidden states，相同硬件下batch size可以开大；于是能导致训练速度更快 。

我们除以方差来进行 方差标准化(variance-normalized) ，来解决维度分布差异/量纲不同(different variances)的问题。

即如果想求点 x 到某一个聚类中心点 c 的距离，假设是两个维度，则是减去均值，除以标准差：

d(x,c)=(x1−c1)2σ12+(x2−c2)2σ22d(x, c)=\sqrt{\frac{\left(x_1-c_1\right)^2}{\sigma_1^2}+\frac{\left(x_2-c_2\right)^2}{\sigma_2^2}} d(x,c)=σ12​(x1​−c1​)2​+σ22​(x2​−c2​)2​​

注意，横纵两个分布是独立的，也就是说它们之间没有covariance 。

Correlation 相关性：
variance-normalized 解决了维度权重不一样的问题。但是如果两个维度分布之间有关联，怎么解决？


如果维度间不独立同分布，样本点一定与欧氏距离近的样本点同类的概率更大吗？
图中，红色点和绿色点距离中心点的距离是一样的，但是直观看起来，红色点是离群点。也就说，就算已经进行了variance-normalized， 相对距离还是不变的。因此，我们必须考虑加上correlation.

  1. 将变量按照主成分进行旋转，让维度间相互独立。

Long-range Language Modeling with Self-retrieval

推理过程：在长文本生成过程中，模型进行自举检索

  1. LM 编码当前 a recently generated text chunk ，得到 query representation
  2. 然后检索出早先生成的 earlier chunks in the document
  3. 然后使用 retrieved chunks representation 融入 LM 继续进行生成 next target chunk

目标： retrieve chunks that increase the probability of the next chunk

LightGBM（Light Gradient Boosting Machine / LGBM）

进一步提出了一系列的 数据压缩 和 决策树建模流程 的优化策略；
解决GBDT算法框架在处理海量数据时计算效率低下的问题，以牺牲极小的计算精度为代价，将GBDT的计算效率提升了近20倍；

算法上：

  • 单边梯度采样法（Gradient-based One-Side Sampling, GOSS）；
    样本采样；只采样梯度大的样本和少部分小梯度样本。
  • 互斥特征捆绑 （Exclusive Feature Bundling, EFB）；
    特征降维；合并不同时为空的特征，在尽可能小的特征损失下，降低要用的特征数量。
  • 采用了直方图算法（Histogram algorithm）和直方图作差的计算方法；
    将占内存空间大的连续特征，以分桶的形式，转为离散特征；
    直方图的叶子节点，有一个可以通过作差的方式得出。

决策树训练上：

  • 决策树生长策略由Level-wise变为Leaf-wise；

工程上

  • 特征并行，数据并行，投票并行。

数据压缩策略

背景：

  • GBDT 在进行每一棵树的训练时，需要考虑全部特征，计算信息增益，从而找到最佳切分点，即「扫描全部数据来决定切分点」。这种方法十分精确，但是计算复杂度很高。
  • 为了降低计算复杂度，通常的做法是「下采样」和「降维」，简单的欠采样会造成方差变大，模型不稳定；而「降维」会导致信息丢失，所以只适用于「特征冗余/ feature redundancy」的情况。

改进：

  1. 特征压缩： 连续变量离散化，超参数「箱体的数量」控制 等宽分箱
  2. 特征压缩： 带入离散特征和离散后的连续变量进行离散特征捆绑（合并）降维：：互斥特征捆绑（Exclusive Feature Bundling, EFB）算法
    1. 独热编码的逆向过程，通过把互斥的特征捆绑到一起来实现降维，克服传统降维方法带来的信息大量损耗的问题。
    2. 输入EFB进行降维的特征，即包括原始离散特征，也包括第一阶段连续变量离散化之后的特征。
  3. 下采样： 在每次构建一颗树之前进行样本下采样：在每次迭代（也就是每次训练一颗决策树模型）的时候，围绕训练数据集进行 基于梯度的单边下采样（Gradient-based One-Side Sampling, GOSS） 。来大幅压缩数据，但同时又不会导致信息的大量损失。

等宽分箱

主要讨论连续形变量，其实离散形式的变量也可以进行分桶，见下文 。

首先计算连续型变量的值域，然后通过 max_bin 超参数，等间距地划分为 max_bin 个箱子，最后把原本的连续变量的值替换为箱体的 id 。
同时 LightGBM 对特征的每个桶进行梯度（一阶和二阶梯度）累加和个数统计，即存储了整个桶的导数和，后续直接应用于特征划分点采样。

区分 XGBT ：

  • XGBT 采样分位数分箱，即每个箱子内部样本权重和差不多，目的是为了减少特征划分点的数量 。
    LGBT 是直接通过连续性变量的最大最小值等间距地确定 bin 的范围，目的是为了把原本连续的特征离散化 。
  • XGBT 需要首先根据特征值进行排序；而 LGBT 不需要，只需要找到最小最大值来进行 。
  • XGBT 分桶后，仍然存储原始数据的浮点数据；而 LGBT 存储的是映射后的桶 id 。

直方图算法并非没有缺点，由于histgoram会把原本连续的特征离散化，这就在一定程度上损失了特征的信息，使得节点分裂时找到的分割点不是很精确，因此也会对结果产生一定影响（训练误差没有 pre-sorted 好）。但好在LightGBM本身就是弱学习器，这部分的信息损失在梯度提升（Gradient Boosting）的框架下并其没有带来太大的影响，而且从某种层度上还带来了正则化的效果，从而降低了模型整体的过拟合风险。

  1. 内存占用降低了7倍。对比类似于XGBoost的Pre-sorted 算法，由于其是基于32位浮点型来存储原始特征，而且每一个特征还需要相同内存大小的索引空间，因此需要(2 * #data * #features * 4Bytes) 的空间。但直方图算法只需要 (#data * #features * 1Bytes) 的内存消耗，仅为 pre-sorted 算法的 1/8。
    考虑直方图算法仅需要存储 feature bin value (离散化后的数值)，不需要原始的 feature value，也不用排序，而 bin value 用 uint8(256 bins) 的类型一般也就足够了。
  2. 避免了XGBoost中的cache miss问题。在XGBoost中，由于特征的梯度值在存储上并不连续，因此节点分裂取数据时，经常会发生cache miss的问题，尽管XGBoost后续也提出了优化方法。但在histogram 算法里，不同特征访问梯度的顺序是一样的，可以提前把梯度存在连续的数组中，让不同特征访问的时候都是连续的，不会产生cache miss的问题。
  3. 大幅度减少了决策树的节点分裂次数。对于一个特征，pre-sorted 需要对每一个不同特征值都计算一次分割增益，而histogram只需要计算 #bin (histogram 的横轴的数量) 次。
  4. 数据并处理时，可以大幅度减少通信开销。在数据并行的时候，用 histgoram 可以大幅降低通信代价。用 pre-sorted 算法的话，通信代价是非常大的（几乎是没办法用）。

如何进行分桶操作的呢？如何进行分桶操作的呢？

  • 连续性特征：
    1. 对特征值去重后进行排序（从大到小）并统计每个特征值出现的次数 counts。
    2. 取 max_bin 和 distinct_value.size 中的较小值作为 bins_num
    3. 计算每个桶可以分到的平均样本个数 mean_bin_size，特征取值数 distinct_value.size 比max_bin 数量少，直接取distinct_values的中点作为桶间分割点，即无需分桶。反之则需要分桶，也就是说可能存在几个特征值同分于一个桶中（多特征取值公用一个桶），但是有一点就是当该特征取值的计数值大于平均值 mean_bin_size 时，该特征取值需要单独分桶，所以需要标记出符合该特点的全部特征，之后对不符合的重新计算 mean_bin_size。
    4. 然后对于去重后的特征取值进行遍历操作，如果当前的特征需要单独成桶、或者当前桶中个特征计数超过了 mean_bin_size、或者下一个特征是需要独立成桶的，那么当前的特征值将作为当前桶的上界，下一个桶的下界，也就是说需要本步需要结束当前桶的构建，下一步需要建立新的桶了。
  • 离散性特征：
    1. 首先对特征取值按出现的次数排序（大到小）。
    2. 取前 min(max_bin, distinct_values_int.size()) 个特征做特征值到桶之间的映射（这样可能会忽略一些出现次数较少的特征取值），也就是取 max_bin 和 distinct_value.size 中的较小值作为 bins_num。
      3。 然后用 bin_2_categorical_（vector类型）记录桶对应的特征取值，以及用categorical_2_bin_(unordered_map类型) 将特征取值对应的桶。

直方图优化算法

XGBT 加权分位求特征划分点的弊端：

  1. 需要对每个特征按特征值进行排序。
  2. 由于对特征进行了排序，但梯度并未排序，所以梯度值的获取属于随机内存访问。（内存利用率不高
  3. 预排序方法（pre-sorted）：首先，空间消耗大。这样的算法需要保存数据的特征值，还保存了特征排序的结果（例如排序后的索引，为了后续快速的计算分割点），这里需要消耗训练数据两倍的内存。其次时间上也有较大的开销，在遍历每一个分割点的时候，都需要进行分裂增益的计算，消耗的代价大。
  4. 对cache优化不友好。在预排序后，特征对梯度的访问是一种随机访问，并且不同的特征访问的顺序不一样，无法对cache进行优化。同时，在每一层长树的时候，需要随机访问一个行索引到叶子索引的数组，并且不同特征访问的顺序也不一样，也会造成较大的cache miss。

LGBT 使用Histogram算法，先把连续的浮点特征值离散化成k个整数，同时构造一个宽度为k的直方图。遍历数据时，根据离散化后的值作为索引在直方图中累积统计量，当遍历一次数据后，直方图累积了需要的统计量，然后根据直方图的离散值，遍历寻找最优的分割点。

互斥特征捆绑降维（Exclusive Feature Bundling，EFB）

具体的 EFB 算法，类似于 one-hot 编码的逆向变换，即将多组 one-hot feature 转换为一个 numeric feature 。

操作
选定几个特征，得到特征组 / EFB（Exclusive Feature Bundling），特征组内部的特征增通常是互斥的，即特征之间不可能同时为 1 ，所以称其为 “Exclusive Feature”（独占特征）。
然后将其不同的组合转化为不同的 ID 表示 。类似于使用一个代表性特征（Exclusive Feature）负责代表整个组。

但是其实严格互斥的特征组还是比较少的，所以这里引入一个超参数： 最大冲突比例（conflict_rate）
首先定义冲突：即存在两条数据同时取得了非零值，然后计算 冲突比例（conflict_rate） ，为「冲突数量」/「非全零样本数量」。

如果 冲突比例（conflict_rate） 小于 最大冲突比例（conflict_rate） ，则我们可以进行捆绑。

显然，超参数 最大冲突比例（conflict_rate） 越小，对冲突的容忍比例越低，最终特征数量也就越多，计算量越大，精度越高。

真实场景操作

真实实现时，会将特征捆绑问题转化为一个「图最小着色数量的问题」，即将每一个特征看作看作图上的一个顶点，如果特征之间存在冲突，那么就会有一个无向边进行连接，边的权重就是「冲突比例」。然后考虑，如何使用尽量少的颜色对顶点进行着色，使得相邻顶点之间的颜色不相同。
不互斥/有冲突的特征对应的顶点之间有边，所以颜色不同，也就是没有办法分到同一个特征组中。
相同颜色的点其实就是互斥的特征，可以被分到同一个特征组中。

流程为：

  1. 计算特征之间的冲突值
  2. 构建无向图，根据超参数「大冲突比例」删除边
  3. 进行着色：从「度」更大的点进行着色，然后根据贪心算法，依次进行 BFS 。
  4. 将同一颜色的特征划分到同一个特征组中。
  5. 同一个特征组进行特征捆绑：
    1. 首先找到一个「主特征」，根据主特征的最大值设置一个 offset 值。
    2. 对「合并特征」，如果值非零，则特特征值加一个 offset 值。
    3. 将「主特征」和「合并特征」进行相加。

特征合并策略
现在已经将特征合并在一起了，下一个问题就是如何得到这一个组合的箱体的 id 值，即将多个 numeric vector 合并为一个 numeric vector，作为这个箱体的 id。

很简单，那么当多个特征合并到一起时，EFB算法只需要要给不同的特征分配不同的取值空间（bins）即可。

举例来说，假设特征A的取值空间为 [0,10)[0, 10)[0,10) ，那么预期捆绑在一起的特征B（假设其bin的取值空间为 [0,20][0, 20][0,20] ），只需要在B原本的bins上都加10就好了。
通过这种方式，EFB算法使得捆绑在一起的特征在取值上不重叠。

LightGBM通过互斥特征捆绑的方式，降低了特征的数量，
使得直方图的构建时间复杂度从 O(#data ∗ #feature)\text{O(\#data ∗ \#feature)}O(#data ∗ #feature) 变为了 O(#data ∗ #bundle)\text{O(\#data ∗ \#bundle)}O(#data ∗ #bundle) 。

为什么 LGBM 不需要 one-hot 编码 / 逆变换可行性

「one hot 编码」本质上是对离散特征更加细粒度的信息呈现，有助于学习。
「one hot 编码」对线性模型（linear/logistic regression）效果很好，理论上对树模型没有用。

使用了one-hot 编码后，当节点划分时，只能做one-vs-rest（如是不是猫，是不是狗，等等）。
当特征的维度比较高时（极端情况下把身份证ID用于训练），这时每个类别上的数量都很少，而这就会带来切分不平衡问题。
切分后的零散小空间比较多，当数据量较小时，学到的信息就会变差，从而就会影响模型最终的效果。（本质是因为独热码编码之后的特征的表达能力较差的，特征的预测能力被人为的拆分成多份，每一份与其他特征竞争最优划分点都失败，导致该特征最终得到的重要性会比实际值低。）

原因：

  • LGBM 的算法设计就是为了处理海量高维数据，独热编码会引入高维度的稀疏特征，这会增加模型的复杂性和计算成本。
  • 梯度提升树天然适用于处理混合类型的特征，可以在训练过程中自动选择适合的分割点，而不需要将离散特征扩展为独热编码。
  • LightGBM使用了一种基于直方图的分桶策略，可以有效地处理离散特征。它会将数据集中的特征值进行分桶，从而降低了内存使用和计算成本，无需显式地将离散特征转换为独热编码。

基于梯度的单边下采样（Gradient-based One-Side Sampling, GOSS）

目的是为了减小训练过程中的样本数量，以减小内存和计算量。如何选择采样哪些样本就成了关键。

样本选择：
在执行优化算法的过程中，每个样本都有一个对应的梯度计算结果，如果某条样本梯度绝对值较小，则说明这条样本已经被正确的分类或者预测结果和真实结果非常接近，后续训练过程中会造成资源浪费。

在每次构建新的树之前进行抽样：

  1. 将全部样本按照 梯度绝对值 大小进行降序排序

  2. 抽取梯度绝对值最大的前 a%a\%a% 的样本，然后把其他样本都视作小梯度样本，并从这些小梯度样本中（不放回）随机抽取 b%b\%b% 个样本。
    默认情况下，这两个超参数 top_rate a%a\%a% 和 other_rate b%b\%b% 是所有树共享的。
    而如果这两个参数同时较大，则会增加模型训练复杂度，增加模型训练时间。
    一般来说top_rate越大other_rate越小，则模型过拟合风险就越大，反之则模型会欠拟合。

  3. 这些大梯度样本和随机抽取的小梯度样本，就构成了接下来模型训练的数据集。

  4. 在决策树生长过程中，「小样本的梯度和二阶导数」会乘以一个「膨胀系数」 1−ab\dfrac {1-a}{b}b1−a​ ，再和「大样本的梯度和二阶导数」求总和，来计算数据集的梯度，来指导树木的生长训练。
    目的是为了尽可能还原原始数据集的完整梯度，提高模型精度。

决策树训练

Leaf-wise tree growth的叶子节点优先的决策树生长策略

其他大多数决策树算法和集成算法来说（CART, XGB），树都是一次生长一层，也就是所谓的 Level-wise tree growth ；

优点：具有天然的并行能力，即同一层的节点可以并行地向下进行分裂。
缺点：有很多不必要的分裂产生，造成更多的计算代价。

Sentence-T5

三种得到句向量的方法 ：

  1. Encoder-only first ：利用T5 最后一层encoder第一个token的特征输出作为句向量。因为不是 cls ，所以不是很靠谱
  2. Encoder-only mean ：利用T5最后一层encoder所有token的特征输出平均值作为句向量。
  3. Encoder-Decoder first ：利用T5最后一层 decoder 的第一个 token 的输出作为句向量。
    为了获得这个 token 的输出，需要将 input 文本输入 encoder，还需要将代表start的 token 喂给 decoder。
    然后用 decoder 生成的 token representation + MLP + Normalization 作为句子的表征。

实验发现 2. 3. 效果比较好。

训练时使用了双塔结构，左右两边使用一个 backbone 。
方法使用 in-batch contrastive learning 。
采用了两阶段的训练方式，第一个阶段现在「通用领域的问答数据训练（Community QA）」,第二阶段才在「人工标注的数据（NLI）」上训练。

实验中训练过程使用了巨大的batch size，第一阶段的batch size是2048，第二阶段的batch size是512，远超simcse实验中的最佳参数配置64。

Consistency-based filter

提出了一种一致性过滤（Consistency-based filter）的方法：

  1. 先让模型在13亿的噪声数据上进行训练 。
  2. 利用训练好的模型给每个数据对（q,p）排序，候选池里包括一百万个passage，如果p能落在top_k个候选结果中，就保留该数据对，否则则移除该数据对。
    这种技术的直觉来自神经网络的记忆行为：当在噪声数据集上进行训练时，神经网络倾向于先记忆干净的标签，然后逐渐过拟合噪声标签。

希望模型的预测能跟训练数据保持一致。在文中作者将k设置为2，通过这种方式最终获得包括2.7亿文本对数据的数据集CCPairs。

目前检索模型的预训练有两种策略：

  1. self-contrastive learning：受限于数据增强的质量，并且需要采用非常庞大数量的的负样本。

如何提高LLMs的文本表征(Text Embedding)能力?

为什么主流方法不用直接用大模型进行文本向量表征？
效果不理想。现在大模型的训练都是基于文本续写形式来训练的，损失函数主要预测下一个Token是否准确，而不是判断整个句子表征的好坏。因此直接用当前的大模型来对文本进行表征，效果并不理想。但如果将大模型以文本表征为目的的训练，也就用point-wise、pair-wise、list-wise等方法训练大模型，相信大模型文本向量表征一定更好。

《Improving Text Embeddings with Large Language Models》:

  1. 构造高质量训练数据: 直接挖掘大模型内部存储的知识内容
    1. 通过GPT4模型进行头脑风暴，生成固定个数的检索任务列表
    2. 通过GPT4模型根据上一步生成的任务定义、Query类型、Query长度、Query清晰度、文档片段长度、语言等生成具体示例。
  2. 文本向量表征时写好提示词
    将数据末尾添加一个[EOS]标记，文本经过大模型后，将[EOS]标记对应的向量作为整个文本的句子向量。并采用InfoNCE作为损失函数，负例采用Batch内负例和一个难负例。
  3. 选对底座大模型
    DeepSpeed ZeRO-3 + Lora 微调 mistral-7b
    基座模型的选择比较重要，选择Llama2-7B要比Mistral-7B低1.6%，说明底座模型越好，预训练数据效果越好
