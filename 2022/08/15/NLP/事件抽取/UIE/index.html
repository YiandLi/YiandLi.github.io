UIE v1

Paper：ACL2022 - Unified Structure Generation for Universal Information Extraction
Code：universal-ie/UIE

Abstract and Introduction

Information Extraction suffers from:

  • varying targets
  • heterogeneous structures
  • demand specific schemas

So, authors propose UIE, a unified text-to-structure generation framework, which:

  • unify different IE structures(answer) into a uniform representation via a structured extraction language (SEL)
  • generate targeted structures(answer) via a schema-based prompt mechanism – structural schema instructor (SSI)
    , which controls what to spot, what to associate, and what to generate in UIE.
  • captures the common IE abilities via a large-scale pre-trained text-to-structure model

To unify all IE tasks:

  1. All IE tasks can be modeled as text-to-structure transformations,
    an entity is a named span structure, an event is a schema-defined record structure.

  2. and These text-to-structure transformations can be further decomposed into several atomic transformation operations:

    1. Spotting: locates the desirable spans
    2. Associating: connects spans by assigning them with semantic roles

In this way:

  1. different IE tasks can be decomposed into a sequence of atomic text-to-structure transformations
  2. all IE models share the same underlying spotting and associating abilities

Structured Extraction Language

To uniformly encode heterogeneous extraction answers
by encoding different IE structures via the spotting-associating structure.

as shown above, each SEL expression contains three types of semantic units:

  2. Asso Name: there is another span that is of this association to its upper-level spotted span
  3. Info Span: the text span/value of SPOTNAME or ASSONAME keys

Plus, : means the mapping from InfoSpan to its spotting or associating names
( and ) show the hierarchical structure between the extracted information

Structural Schema Instructor


One Challenge: Since different IE tasks have different schemas, how to adaptively control which information to generate ?

Authors propose a structural schema instructor (SSI), a schema-based prompt mechanism that controls which kinds of information need to be spotted and associated.

UIE takes the given structural schema instructor/prompt (s) and the text sequence (x) as input, and generates the linearized SEL (y):

y=UIE⁡(s⊕x)y=\operatorname{UIE}(s \oplus x) y=UIE(s⊕x)

x=[x1,…,x∣x∣]x=\left[x_{1}, \ldots, x_{|x|}\right]x=[x1​,…,x∣x∣​] : text sequence
s=[s1,…,s∣s∣]s=\left[s_{1}, \ldots, s_{|s|}\right]s=[s1​,…,s∣s∣​] : structural schema instructor
y=[y1,…,y∣y∣]y=\left[y_{1}, \ldots, y_{|y|}\right]y=[y1​,…,y∣y∣​] : a SEL sequence


SSL Contains three types of token segments:

  1. SPOTNAME: the targeted spotting name
  2. ASSONAME: the targeted association name
  3. Special Symbols: ([spot], [asso], [text]), added before each SPOTNAME, ASSONAME, and input text sequence.

The entire input for UIE is in the form of:

s⊕x=[s1,s2,…,s∣s∣,x1,x2,…,x∣x∣]=[[spot⁡],…[spot⁡]…,[asso⁡],…,[asso⁡]…,[ text ],x1,x2,…,x∣x∣]\begin{aligned} s \oplus x=& {\left[s_{1}, s_{2}, \ldots, s_{|s|}, x_{1}, x_{2}, \ldots, x_{|x|}\right] } \\ =& {[[\operatorname{spot}], \ldots[\operatorname{spot}] \ldots,} \\ & {[\operatorname{asso}], \ldots,[\operatorname{asso}] \ldots, } \\ & {\left.[\text { text }], x_{1}, x_{2}, \ldots, x_{|x|}\right] } \end{aligned} s⊕x==​[s1​,s2​,…,s∣s∣​,x1​,x2​,…,x∣x∣​][[spot],…[spot]…,[asso],…,[asso]…,[ text ],x1​,x2​,…,x∣x∣​]​


Advantage:

  1. effectively guide the SEL generation
  2. adaptively control which to spot, which to associate, and which to generate
    --> the same model do different tasks on the same sentence

Structure Generation

Use an end-to-end/encode-to-decode structure for this text-to-SEL process.

Encoder is a Transformer encoder:

H=Encoder⁡(s1,…,s∣s∣,x1,…,x∣x∣)\mathbf{H}=\operatorname{Encoder}\left(s_{1}, \ldots, s_{|s|}, x_{1}, \ldots, x_{|x|}\right) H=Encoder(s1​,…,s∣s∣​,x1​,…,x∣x∣​)

Decoder is an auto-regressive transformer decoder. At step iii:

yi,hid=Decoder⁡([H;h1d,…,hi−1d])y_{i}, \mathbf{h}_{i}^{d}=\operatorname{Decoder}\left(\left[\mathbf{H} ; \mathbf{h}_{1}^{d}, \ldots, \mathbf{h}_{i-1}^{d}\right]\right) yi​,hid​=Decoder([H;h1d​,…,hi−1d​])

When the decoder predict the token <eos>, the prediction process is over.

Pre-training and Fine-tuning

datasets and training example:

  • the text-structure parallel dataset Dpair\mathcal D_{\text{pair}}Dpair​ , e.g., aligning Wikidata with English Wikipedia
    • each instance consists of token sequence xxx and structured record yyy
    • to pre-train the text-to-structure transformation ability
  • the structure dataset Drecord\mathcal D_{\text{record}}Drecord​ , e.g., knowledge bases
    • each instance is a structured record yyy
    • to pre-train the structure decoding ability
  • the unstructured text dataset Dtext\mathcal D_{\text{text}}Dtext​ , e.g., raw texts
    • each instance is a plain text in English Wikipedia
    • to pre-train the semantic encoding ability

Pre-training

Text-to-Structure Pre-training

on Dpair={(x,y)}\mathcal D_{\text{pair}}=\{(x,y)\}Dpair​={(x,y)}
target: extract the spot type ss+s_{s+}ss+​ and the associating type sa+s_{a+}sa+​, which is s+=ss+∪sa+s_+=s_{s+} \cup s_{a+}s+​=ss+​∪sa+​

If only use positive schema, the model will only simply remember the triplet in the pre-training data.
So, automatically construct negative schemas for each pair, i.e., first sample negative spots ss−s_{s-}ss−​ and negative association set sa−s_{a-}sa−​, then concatenate meta-schema smeta=s+∪ss−∪sa−s_{meta} = s_+ \cup s_{s-} \cup s_{a-}smeta​=s+​∪ss−​∪sa−​ .
For negative spots and association, the model should predict Null as an answer to the key.

Structure Generation Pre-training

on Drecord\mathcal D_{\text{record}}Drecord​
target:

  • pre-train the ability of generating valid structures defined by SEL and schemas
  • the decoder can capture the regularity of SEL and the interactions between different labels.

LRecord =∑y∈Drecord −log⁡p(yi∣y<i;θd)\mathcal{L}_{\text {Record }}=\sum_{y \in \mathcal{D}_{\text {record }}}-\log p\left(y_{i} \mid y_{<i} ; \theta_{d}\right) LRecord ​=y∈Drecord ​∑​−logp(yi​∣y<i​;θd​)

Retrofitting Semantic Representation

on Dtext\mathcal D_{\text{text}}Dtext​
target: alleviate the catastrophic forgetting of token semantics especially on SPOTNAME and ASSONAME tokens

add span corruption based mask language modeling objective(Raffel et al., 2020) in the pre-training stage:

LText=∑x∈Dtext −log⁡p(x′′∣x′;θe,θd)\mathcal{L}_{\mathrm{Text}}=\sum_{x \in \mathcal{D}_{\text {text }}}-\log p\left(x^{\prime \prime} \mid x^{\prime} ; \theta_{e}, \theta_{d}\right) LText​=x∈Dtext ​∑​−logp(x′′∣x′;θe​,θd​)

x′x^{\prime}x′ is the corrupted source text and x′′x^{\prime \prime}x′′ is corrupted target spans.

Final Pre-training Criteria

L=LPair +LRecord +LText \mathcal{L}=\mathcal{L}_{\text {Pair }}+\mathcal{L}_{\text {Record }}+\mathcal{L}_{\text {Text }} L=LPair ​+LRecord ​+LText ​

For implementation, all pre-training data are represented as triplets.

  • Dpair={(x,y)}\mathcal D_{\text{pair}}=\{(x,y)\}Dpair​={(x,y)}: (ssl, x, sel)
  • Drecord\mathcal D_{\text{record}}Drecord​: (None, None, sel)
  • Dtext\mathcal D_{\text{text}}Dtext​: (None, x′x^{\prime}x′, x′′x^{\prime \prime}x′′)

On-Demand Fine-tuning

Given a labeled corpus Dtask={(s,x,y)}\mathcal D_{task}=\{(s,x,y)\}Dtask​={(s,x,y)}
The fine-tunning teacher-forcing cross-entropy loss is

LFT=∑(s,x,y)∈DTask −log⁡p(y∣x,s;θe,θd)\mathcal{L}_{\mathrm{FT}}=\sum_{(s, x, y) \in \mathcal{D}_{\text {Task }}}-\log p\left(y \mid x, s ; \theta_{e}, \theta_{d}\right) LFT​=(s,x,y)∈DTask ​∑​−logp(y∣x,s;θe​,θd​)

To alleviate the exposure bias (Ranzato et al., 2016; Zhang et al., 2020) of the auto-regressive model during decoding,
the author propose a method called “a Rejection mechanism”:
randomly insert several (SPOTNAME, [NULL]) and (ASSONAME, [NULL]) into the ground-truth SEL with the probability of pϵp_{\epsilon}pϵ​ .


e.g., sentence “Steve became CEO of Apple in 1997”
facility is the negative spot in the schema prompt, so to randomly inject the noise of (facility: [NULL]) into the target record.
In this way, the UIE can effectively learn to reject misleading generation by generating [NULL] token.

Experiments

  • Datasets:
    4 well-representative IE tasks (including entity extraction, relation extraction, event extraction, structured sentiment extraction)
    and their combinations (e.g., joint entity-relation extraction).

Get final result

mapped the generated string-level extraction results to offset-level(start index/end index) for model evaluation:

  • For entity/trigger mentions,
    reconstruct the offset of predicted entity/trigger mentions by finding the matched utterance in the input sequence one by one.
  • For argument mentions in relation and event tasks,
    the nearest matched utterance is assigned as the predicted entity/trigger mention as the predicted offset.

Experiments on Supervised Settings

  1. SEL colum: By modeling IE as text-to-structure generation and encoding with an effective SEL language, UIE provides an effective universal architecture for IE。
  2. UIE colum: The large-scale pre-trained model provides a solid foundation for universal IE.
  3. UIE colum among all the tasks: By universally modeling IE tasks and pre-training using large-scale datasets, UIE can effectively capture, share, and transfer IE abilities.

Experiments on Low-resource Settings

  • Baselines:
    • T5-v1.1-base
    • Fine-tuned T5-base
    • UIE-base w/o SSI
  • six different partitions of the original training sets (1/5/10-shot, 1/5/10% ratio) across 4 tasks
  • repeated each experiment 10 times with different samples and reported their averaged score

  1. By guiding the generation using schema-based prompts, SSI is an effective way for adaptively controlling which to extract.
  2. Our pre-training algorithms can learn general IE ability rather than capture task-specific information.

UIE-USM v2

Paper：ACL2023 -Universal Information Extraction as Unified Semantic Matching

UIE v1 的缺点：
Seq2Seq 的生成模型的黑盒特性，导致无法判断跨任务或跨模式的迁移在什么情况下会成功 or 失败。

本论文提出了统一语义匹配框架（Universal Information Extraction，USM）对各种信息提取任务进行统一建模，主要包含两个部分：

  1. Structuring：从文本中抽取目标结构中 标签未知 的基本子结构，即不知道标签，只抽取可能出现的文本链接结构。
  2. Conceptualizing：将上述连续片段和成对的子结构链接为相应的目标语义概念，即给抽出的结构打上标签。

这两个操作都可以使用 semantic matching 进行建模：

  1. structuring 任务：原始序列 utterances or substructures 的链接任务
  2. Conceptualizing 任务：target semantic labels 和 utterances or substructures 的链接任务。

Schema-Text Joint Embedding

输入：原始输入文本 ttt ，模版序列 lll ，类似 UIE 中的 SEL，即将标签拼接为前缀文本和输入文本进行拼接。

在构造模版序列时，使用一种 [L] marker：

  1. [LM] ：single mention，对应 entity types 和 event trigger types 。
  2. [LP] ：association pair，对应 relation types 和 event
    argument types。

使用 Transformer-encoder 进行编码：

H=[h1,h2,…,h∣l∣+∣t∣]=Encoder⁡(l1,l2,…,l∣l∣,t1,t2,…,t∣t∣,M)\mathbf{H}= \left[\mathbf{h}_1, \mathbf{h}_2, \ldots, \mathbf{h}_{|l|+|t|}\right]= \operatorname{Encoder}\left(l_1, l_2, \ldots, l_{|l|}, t_1, t_2, \ldots, t_{|t|}, \mathbf{M}\right) H=[h1​,h2​,…,h∣l∣+∣t∣​]=Encoder(l1​,l2​,…,l∣l∣​,t1​,t2​,…,t∣t∣​,M)

M\mathbf{M}M 是一个 mask 矩阵

Token-Token Linking for Structuring

H2L，H2H，T2T

  1. 抽取 Utterance（entity，event trigger），直接定义为 inner span head-to-tail (H2T) linking ，如果是单个的 token，则自己连接自己，如果是连续的 token 片段，则 start token 指向 end token。

  2. 抽取 Association pair（relation，event argument），定义为 head-to-head (H2H) and tail-to-tail (T2T) linking ，即两种形式的链接，subject start token 指向 object start token，subject end token 指向object end token。

抽取方式为：

sTTL(ti,tj)=FFNNTTLl(hti)TRj−iFFNNTTLr(htj)\mathbf{s}_{\mathrm{TTL}}\left(t_i, t_j\right)=\mathrm{FFNN}_{\mathrm{TTL}}^l\left(\mathbf{h}_t^i\right)^T \mathbf{R}_{j-i} \mathrm{FFNN}_{\mathrm{TTL}}^r\left(\mathbf{h}_t^j\right) sTTL​(ti​,tj​)=FFNNTTLl​(hti​)TRj−i​FFNNTTLr​(htj​)

类似 biaffine， Rj−i\mathbf{R}_{j-i}Rj−i​ 为旋转位置嵌入，可以有效地将相对位置信息注入到对应的结构中。

Label-Token Linking for Utterance Conceptualizing

给定 label token embedding h1l,h∣l∣l\mathbf{h}^l_1, \mathbf{h}^l_{|l|}h1l​,h∣l∣l​ 和 text token embeddings h1t,h∣t∣t\mathbf{h}^t_1, \mathbf{h}^t_{|t|}h1t​,h∣t∣t​

进行 label-token linking (LTL) 预测任务，有两种链接类型：

  1. 实体类型-实体字段 链接: every single mention -> entity type
  2. 关系/谓词类型-object字段 链接: predicate type -> object candidate

这里同样用两种标签：

  1. label-to-head (L2H) linking operation：label head 2 entity head
  2. label-to-tail (L2T) linking operation：label tail 2 entity tail

sLTL(li,tj)=FFNNLTLlabel(hil)TRj−iFFNN⁡LTLtext(hjt)\mathbf{s}_{\mathrm{LTL}}\left(l_i, t_j\right)=\mathrm{FFNN}_{\mathrm{LTL}}^{\mathrm{label}}\left(\mathbf{h}_i^l\right)^T \mathbf{R}_{j-i} \operatorname{FFNN}_{\mathrm{LTL}}^{\mathrm{text}}\left(\mathbf{h}_j^t\right) sLTL​(li​,tj​)=FFNNLTLlabel​(hil​)TRj−i​FFNNLTLtext​(hjt​)

Token-Label Linking for Pairing Conceptualizing

subject字段-关系/谓词类型 链接
同样是使用两种标签 head-to-label (H2L) 和 ail-to-label (T2L) operations。

sTLL(ti,lj)=FFNN⁡TLLtext (hti)Rj−iFFNN⁡TLLlabel(hlj)s_{T L L}\left(t_i, l_j\right)=\operatorname{FFNN}_{T L L}^{\text {text }}\left(\mathbf{h}_t^i\right) \mathbf{R}_{j-i} \operatorname{FFNN}_{T L L}^{l a b e l}\left(\mathbf{h}_l^j\right) sTLL​(ti​,lj​)=FFNNTLLtext ​(hti​)Rj−i​FFNNTLLlabel​(hlj​)

Schema-constraint Decoding for Structure Composing

使用一种 模版限制的解码策略 a schema-constraint decoding algorithm

  1. 解码由TTL操作提取的 实体文本 和 主客体pair文本
    例如：{“Monet”, “Paris”, “France”, (“Monet”, “Pairs”), (“France”, “Pairs”)};
  2. 通过LTL操作解码 实体类型 和 关系/谓词-object
    例如： {(person,“Monet”), (country, “France”), (birth place, “Paris”), (capital, “Paris”)};
  3. 利用TLL操作解码 subject-关系/谓词
    例如： {(“Monet”, birth place), (“France”, capital)}

这三种解码过程之间不相互依赖，所以支持并行，提升计算效率。

  1. 最后得到完整的实体和关系：
    {(“Monet”, birth place, “Paris”) and (“France”, capital, “Paris”)}

Learning from Heterogeneous Supervision

将异构监督信号链接转化为 <text, token pairs><\text{text, token pairs}><text, token pairs> 的形式进行预训练，token pairs 为标注链接 (TTM, LTM, TLM)。

本文收集了三种不同的监督数据用于USM的预训练：

  • 任务数据 Dtask \mathcal{D}_{\text {task }}Dtask ​ ：来自信息抽取任务的标注数据，即数据样本都有一个 golden answer 。
  • 远程监督数据 Ddistant \mathcal{D}_{\text {distant }}Ddistant ​ ：数据样本来自文本和知识库对齐。
  • 间接监督数据 Dindirect \mathcal{D}_{\text {indirect }}Dindirect ​ ：数据样本来自其他相关的NLP任务，主要使用机器阅读理解的数据，将（问题-文档-答案）实例中问题作为标签模式，文档作为输入文本，答案作为提及。

Learning function

为了缓解数据稀疏性的问题，作者使用类别不平衡损失函数：

MMM 为链接类， m+,m−m^{+},m^{-}m+,m− 分别表示链接对和非链接对。
