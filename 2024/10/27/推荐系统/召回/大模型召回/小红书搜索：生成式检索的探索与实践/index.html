小红书搜索：生成式检索的探索与实践

Posted on 2024-10-27 In Search & Recommendation System , LLM-RQ
Symbols count in article: 2.1k Reading time ≈ 14 mins.

Blog: https://mp.weixin.qq.com/s/EB-uMwJT2SAKUaKSG1tf3g
Paper：Generative Dense Retrieval: Memory Can Be a Burden

Intro

密集检索 DR - 双塔模型

本质是度量学习

优点：

  1. 快速高效地更新索引
  2. 线上服务压力小，一次行建库可以顶一天。
  3. 当候选文档库动态更新（增改删）时，可高效更新索引。
  4. 可以直接看到全局 Doc

缺点：

  1. 无法进行深度交互，效果不如单塔模型。
  2. 查询和文档间具有一对多的特性，候选文档间的语义关联可能较远，而查询想召回所有候选文档则要求它们在语义空间接近。
    也就是只能查到最近邻的 Doc ，但是有时候需要语义没有那么匹配但是更有效的 Doc 。

生成式检索 GR - 生成模型

生成式检索：给定一个 Query，生成式检索生成候选 Doc ID。 Doc ID 可以是单个值或者一个 ID 序列。
本质是记忆机制：让模型记忆 Doc ID，然后泛化到其他新的 Query 。

构造训练数据：对 query 进行处理，为每篇笔记生成一堆 query 来表示笔记，将真实 query 和生成 query 集合作为输入，生成对应的笔记 ID。

优点：

  1. 小规模场景下，生成式检索展现出了优于密集检索的召回性能，因为记忆池没那么大，并且缓解了长尾分布的问题。
  2. 在解码过程中隐式实现深度交互，效果应该更好。
  3. 文档被赋予了独立的 ID，避免了密集检索存在的 one to many 情况
  4. 可以利用 LLM 作为 Backbone，有效利用预训练语言模型的语义理解能力。

缺点：

  1. 构造训练数据寻在问题：没办法看到整个文档，对候选文档中具体的、细粒度的信息无法有效（区分性）记忆。
    体现在：生成式检索模型在前面几位错误率较低，但越往后错误率升得特别快。
  2. 候选文档规模增大会导致性能显著下降：当候选文档规模增大时，性能会显著下降。原因是 模型参数有限，记忆容量有限 ，当候选文档容量超过模型记忆容量时，记忆出现瓶颈，性能下降明显。
  3. 动态更新动态更新成本太高，候选文档更新时可扩展性差，每次都需要重新训练一个模型。
    如果使用增量训练的方法进行更新，容易出现灾难性遗忘问题，即忘记了前面训练时的 QD pair 内容。
    验证实验，将数据集均分为两份，先用一份子集训练 NCI 模型，然后更新候选集不重新训练模型，测试在不同查询集上的表现，发现候选集扩充后召回率下降。
  4. 自回归解码范式导致时间和计算开销大

本文 - 互补的GDR

步骤：

  1. Corse「Inter-cluster Matching」：利用生成式检索的记忆能力记住 high level 的语义类 ID ，即生成文档簇的ID。
  2. Fine「Intra-cluster Matching」：到细粒度的地方用密集检索做匹配 ，即在簇内进行匹配。

Method

给定 Query qqq ，需要从全体 Doc 集合 Dl\mathcal{D}_lDl​ 中检索得到候选子集 Dc\mathcal{D}_cDc​ 。

模型整体是三个组件，一个 Query Encoder，一个生成 CID 的自回归 Doc Decoder，一个用于检索的 Doc Encoder。

Inter-cluster Matching

使用一个 Transformer 结构（T5-base Encoder + randomly initialized PAWA decoder），首先 Encoder 编码 qqq 得到 eq1:∣q∣∈Rde_q^{1:|q|} \in \mathbb{R}^deq1:∣q∣​∈Rd 和 CLS Embedding rqr_qrq​ 。

然后 Decoder(参数 thetathetatheta) 输出 Cluster ID ，即：

finter :q→CID1:kf_{\text {inter }}: q \rightarrow \mathrm{CID}^{1: k} finter ​:q→CID1:k

CID 的生成概率为：

p(CID⁡i∣eq,rq,θ)=∏j=1∣CIDi∣p(CID⁡ji∣eq,rq,CID⁡<ji,θ)p\left(\operatorname{CID}^i \mid e_q, r_q, \theta\right)=\prod_{j=1}^{\left|\mathrm{CID}^i\right|} p\left(\operatorname{CID}_j^i \mid e_q, r_q, \operatorname{CID}_{<j}^i, \theta\right) p(CIDi∣eq​,rq​,θ)=j=1∏∣CIDi∣​p(CIDji​∣eq​,rq​,CID<ji​,θ)

直接将概率作为每个聚类的分数 $$S_{\text {inter }}\left(q, \mathrm{CID}^i\right)$$ 。

所以训练损失为传统的自回归CE损失：

LInter =−log⁡p(CID⁡(d+)∣EQ(q),θDC)\mathcal{L}_{\text {Inter }}=-\log p\left(\operatorname{CID}\left(d^{+}\right) \mid E_Q(q), \theta_{D_C}\right) LInter ​=−logp(CID(d+)∣EQ​(q),θDC​​)

Intra-cluster Matching

需要从上面得到 Clusters 中挑出最近邻的 Doc ：

fintra :CID1:k→d1:kf_{\text {intra }}: \mathrm{CID}^{1: k} \rightarrow d^{1: k} fintra ​:CID1:k→d1:k

这里比较简单，双塔模型，还是 Doc 建库，得到所有 Doc 的表征 ：

rd1:∣Dl∣∈Rdr_d^{1:\left|\mathcal{D}_l\right|} \in \mathbb{R}^d rd1:∣Dl​∣​∈Rd

然后和 Query 作相似度计算，用 Sigmoid 压到 0-1 之间：

Sintra (q,di)=Sigmoid⁡(sim⁡(rq,rdi))S_{\text {intra }}\left(q, d^i\right)=\operatorname{Sigmoid}\left(\operatorname{sim}\left(r_q, r_d^i\right)\right) Sintra ​(q,di)=Sigmoid(sim(rq​,rdi​))

损失函数使用了 In-Batch NCE Loss，同时为了提高分辨率，同时强化相同 CID 中的其他 Doc 作为负样本：

LIntra =−log⁡esim⁡(q,d+)γ∗∑d∈Naesim⁡(q,d)+∑d∈Nresim⁡(q,d)\mathcal{L}_{\text {Intra }}=-\log \frac{e^{\operatorname{sim}\left(q, d^{+}\right)}}{\gamma * \sum_{d \in N_a} e^{\operatorname{sim}(q, d)}+\sum_{d \in N_r} e^{\operatorname{sim}(q, d)}} LIntra ​=−logγ∗∑d∈Na​​esim(q,d)+∑d∈Nr​​esim(q,d)esim(q,d+)​

NaN_aNa​ 为同CID的负样本，仅包含了 d∈CID⁡(d+)d \in \operatorname{CID}\left(d^{+}\right)d∈CID(d+) 。
NrN_rNr​ 为全局负样本，不考虑 CID 来采样。
γ=2\gamma = 2γ=2 用于重点考虑 NaN_aNa​ 。

最终打分

最终 d 的分数需要考虑上述两者：

Soverall (q,di)=Sinter (q,CID⁡(di))+β∗Sintra (q,di)S_{\text {overall }}\left(q, d^i\right)=S_{\text {inter }}\left(q, \operatorname{CID}\left(d^i\right)\right)+\beta * S_{\text {intra }}\left(q, d^i\right) Soverall ​(q,di)=Sinter ​(q,CID(di))+β∗Sintra ​(q,di)

Construction of Memory-friendly CIDs

关于如何构建 Cluster ID，我们希望：

  1. 相同的 query 的 positive Doc 在一个 Cluster 中，也就是有共享前缀。
  2. 可以手动控制 Cluster 数量，尽量和 Model Size 进行对齐。

首先在训练集上 Fine Tune Ernie ，作为编码器 EDE_DED​ 。
训练方法遵从 ADVERSARIAL RETRIEVER-RANKER FOR DENSE TEXT RETRIEVAL 。

算法比较简单，超参数指定一个聚类数量 kkk ，一个簇内文档最大值 ccc 。
不断自顶向下聚类，每次都得到 kkk 个聚类中心，如果其中某个聚类中文数量超过了 ccc 则进行向下聚类。
整体构造出了一个 k 子节点的树，兄弟节点是按照 0～k 进行标识的。

为了满足第二个特性，作者设置 自适应参数 ccc， 以确保总聚类数 ∣CID∣|\mathrm{CID}|∣CID∣ 不会随着 DlD_lDl​ 改变, 方法如下:

c=∣Dl∣Exp⁡(∣CID⁡∣)c=\frac{\left|D_l\right|}{\operatorname{Exp}(|\operatorname{CID}|)} c=Exp(∣CID∣)∣Dl​∣​

其中 Exp⁡(∣CID∣)\operatorname{Exp}(|\mathrm{CID}|)Exp(∣CID∣) 是 ∣CID∣|\mathrm{CID}|∣CID∣ 的期望值，实验中简化设为 5000 。通过该策略, 在不同大小的 DlD_lDl​ 下, 所得的 ∣CID∣|\mathrm{CID}|∣CID∣ 大致保持在相同数量级，从而满足第二个特性。

Training and Inference

Training

给定 QD 对训练集合，

Strain ={(qi,di)∣i∈(1,…,n)}\mathcal{S}_{\text {train }}=\left\{\left(q^i, d^i\right) \mid i \in(1, \ldots, n)\right\} Strain ​={(qi,di)∣i∈(1,…,n)}

使用 DocT5Query 额外生成 5 个 pseudo queries ( 因为这里仅仅需要记忆 CID，所以不需要那么多的负样本足矣 ) 。

为了加速训练，冻结 Doc Embedder EDE_DED​ 。

使用联合损失进行训练：

LGDR=LInter +LIntra \mathcal{L}_{G D R}=\mathcal{L}_{\text {Inter }}+\mathcal{L}_{\text {Intra }} LGDR​=LInter ​+LIntra ​

Inference Phase

  1. 首先 Beam Search 自回归生成 Top-K CIDs 。

  2. 然后每个 CID 内部检索出 top-M 个 Doc ，评分依据仅为 SintraS_{\text{intra}}Sintra​ 。

  3. 最终全局进行打分，依据为 Soverall S_{\text {overall }}Soverall ​ 。

部署时：

  1. 构造 a prefix tree ，仅生成有效 CID 。

    另一种方法是直接计算所有的 CID score / Prob ，但是感觉「前缀树 + beam search」能降低很多复杂度。

  2. 每个 CID 内单独建库

Experiments

落地场景

  1. 在线流式召回

    通常在召回阶段可以默认触发 DR 密集检索的过程，与此同时并行地调用 GDR 和 LLM 检索，从而在利用 GDR 和 LLM 精准的召回效果的同时，避免其高时延的问题。
    时间：用户在看 DR 结果的时候，可以等待 GDR 和 LLM 。
    精度：多次刷新表示当前展现效果不好，所以需要更高精度的 LLM 。

  2. 高频 query 的检索优化

    高频 Query 构建离线的 CID，线上服务时只需要处理 Fine DR 即可。

  3. 搜前、搜中、搜后全流程推词

    推词对应的候选集数量相对较小（百万级），在 GDR 实验设定的候选集数量范围中，当前参数量下 GDR 模型的性能不会受到巨量候选集的影响（小红书笔记库在亿级以上），其能力能够充分发挥。

  4. 未来落地场景一：GDR+LLM

  5. 未来落地场景二：计算效率自适应的检索策略
