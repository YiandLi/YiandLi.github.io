上下文压缩2：LLMLingua

Posted on 2024-03-26 In NLP , LLM Inference
Symbols count in article: 3.3k Reading time ≈ 16 mins.

作者知乎 Blog ： (Long)LLMLingua
作者中文分享： LLMLingua: 压缩prompt构造LLMs的语言

利用经过 Alignment 的 well-trained 的小模型，例如GPT2-small或者LLaMA-7B，来检测和剔除 prompt 中的不重要 token 。

好处：大幅度减少 context length (提升推理速度，降低显存)；同时因为 context 信息密度更高，有机会提升模型的效果 。

LLMLingua

EMNLP 2023 - Microsoft : LLMLingua: Compressing Prompts for Accelerated Inference

Problem：

  1. long prompt and context ：high cost
  2. redundancy and noise ：语言本身就存在冗余：口语，语种；召回更多的噪声文档，噪声文档数量会大幅度降低效果
  3. positional bias ：lost in the middle；context window 越大，selective bias 越大

Hypothesis：

  1. A well-trained smallLM （trained on >1T tokens） can achieve a PPL distribution ranking comparable to that of LLMs.
  2. Tokens with higher PPL are likely to contain more information for LLMs.

xins={xiins}i=1Lins,xdems={xidems }i=1Ldems ,xque={xique }i=1Lque \boldsymbol{x}^{\mathrm{ins}}=\left\{x_i^{\mathrm{ins}}\right\}_{i=1}^{L^{\mathrm{ins}}}, \boldsymbol{x}^{\mathrm{dems}}=\left\{x_i^{\text {dems }}\right\}_{i=1}^{L^{\text {dems }}}, \boldsymbol{x}^{\mathrm{que}}=\left\{x_i^{\text {que }}\right\}_{i=1}^{L^{\text {que }}} xins={xiins​}i=1Lins​,xdems={xidems ​}i=1Ldems ​,xque={xique ​}i=1Lque ​

现在需要将其压缩为 x~={x~i}i=1L~\widetilde{\boldsymbol{x}}=\left\{\widetilde{x}_i\right\}_{i=1}^{\widetilde{L}}x={xi​}i=1L​ ，压缩比为 τ=L~/L,τ∈[0,1]\tau=\widetilde{L} / L, \tau \in[0,1]τ=L/L,τ∈[0,1] 。

压缩目标就是让原始输出和压缩后输出尽量相同，用 KL 散度进行度量：

min⁡x~,τKL⁡(P(x~G∣x~),P(xG∣x))\min _{\widetilde{\boldsymbol{x}}, \boldsymbol{\tau}} \operatorname{KL}\left(P\left(\widetilde{\boldsymbol{x}}_G \mid \widetilde{\boldsymbol{x}}\right), P\left(\boldsymbol{x}_G \mid \boldsymbol{x}\right)\right) x,τmin​KL(P(xG​∣x),P(xG​∣x))

本文提出的整体方法需要三个组件：

  1. a budget controller：进行 demonstration-level filtering，并且调整其他部分的 compression rate
  2. a token-level iterative compression algorithm：动态对三个部分进行 token-level compression
  3. an iterative prompt algorithm：将小模型和黑盒大模型进行对齐

Budget Controller

出发点：

  • Instruction 和 Question 是最重要的，应该分配更低的压缩比；而 demonstration 很可能冗余，所以能使用更高的压缩比。
  • 作者认为 selective filtering 中 token-level compression 很有可能使压缩提示过于琐碎，从而丢失原始提示中的重要信息。
    所以这里采用 sentence-level compression ，保持一定程度的语言完整性，这里使用 document-level compression 。

首先根据预先定义的全局压缩比 τ\tauτ 和其他两个部分的压缩比，计算 demonstration 部分的压缩比：

τdems =τL−(τins Lins +τque Lque )Ldems \tau_{\text {dems }}=\frac{\tau L-\left(\tau_{\text {ins }} L_{\text {ins }}+\tau_{\text {que }} L_{\text {que }}\right)}{L_{\text {dems }}} τdems ​=Ldems ​τL−(τins ​Lins ​+τque ​Lque ​)​

然后进行 demonstration-level prompt compression ，也就是得到 xdems\boldsymbol{x}^{\mathrm{dems}}xdems 的子集 D\mathbf DD ，步骤为：

  1. 使用一个小模型计算每一个 demonstration 的 PPL 。
    这里不去考虑 instruction 和 question，每一个 demonstration 独立计算 PPL 。
  2. 降序选择 top k，不超过 maximum tokens k⋅τdemsLdemsk \cdot \tau_{\mathrm{dems}} L_{\mathrm{dems}}k⋅τdems​Ldems​

最后再去调整 Instruction 和 Question 部分的压缩比：

Δτ=k⋅τdems Ldems −L~DLins +Lque \Delta \tau=\frac{k \cdot \tau_{\text {dems }} L_{\text {dems }}-\widetilde{L}_{\mathcal{D}}}{L_{\text {ins }}+L_{\text {que }}} Δτ=Lins ​+Lque ​k⋅τdems ​Ldems ​−LD​​

kkk 为 granular control coefficient，是一个超参数，实验设置为 2 。
同时，预设的压缩比为 τins =0.85\tau_{\text {ins }}=0.85τins ​=0.85 ， τque =0.9\tau_{\text {que }}=0.9τque ​=0.9 。

Iterative Token-level Prompt Compression

将经过 demonstration-level compression 的 prompt 记作： x′=(xins ,xD,xque )\boldsymbol{x}^{\prime}=\left(\boldsymbol{x}^{\text {ins }}, \boldsymbol{x}^{\mathcal{D}}, \boldsymbol{x}^{\text {que }}\right)x′=(xins ,xD,xque )

将其分为不同的 segment ： S={s1,s2,…,sm}\mathcal{S}=\left\{s_1, s_2, \ldots, s_m\right\}S={s1​,s2​,…,sm​} ，segment size 实验设置为 100 。

作者认为 Selective Context 中一次性直接计算所有 token 的概率值/困惑度可能不太靠谱，因为如果删除了前面 token，那么后面 token 的概率值/困惑度会相应地改变，所以这里动态计算概率值/困惑度，并且进行 token compression 。

也就是维护一个「压缩后 segment 列表」，每一个 segment 根据「压缩后 segment 列表」计算概率值/困惑度，然后压缩，压缩后结果拼接到「压缩后 segment 列表」中。

然后这里利用一个小模型计算每一个 segment 的概率值：

p(s~j)=∏i=1∑kjL~s,kp(s~j,i∣s~j,<i,s~<j)≈∏i=1Ls,j+∑kj−1L~s,kp(sj,i∣sj,<i,s~<j)\begin{aligned} p\left(\widetilde{s}_j\right) & =\prod_{i=1}^{\sum_k^j \tilde{L}_{s, k}} p\left(\widetilde{s}_{j, i} \mid \widetilde{s}_{j,<i}, \widetilde{s}_{<j}\right) \\ & \approx \prod_{i=1}^{L_{s, j}+\sum_k^{j-1} \widetilde{L}_{s, k}} p\left(s_{j, i} \mid s_{j,<i}, \widetilde{s}_{<j}\right) \end{aligned} p(sj​)​=i=1∏∑kj​L~s,k​​p(sj,i​∣sj,<i​,s<j​)≈i=1∏Ls,j​+∑kj−1​Ls,k​​p(sj,i​∣sj,<i​,s<j​)​

也就是使用压缩后的 segments s~<j\widetilde{s}_{<j}s<j​ 计算当前 segment 的概率值 $p\left(\widetilde{s}_j\right) $ ，其实也就有了每一个 token 的动态概率值 。

因为上一步过程中，我们已经有了 prompt 不同部分的 compression ratio，

τsj={τins +Δτ, if sj from xins τdems , if sj from xDτque +Δτ, if sj from xque \tau_{s_j}=\left\{\begin{aligned} \tau_{\text {ins }}+\Delta \tau, & \text { if } s_j \text { from } x^{\text {ins }} \\ \tau_{\text {dems }}, & \text { if } s_j \text { from } x^{\mathcal{D}} \\ \tau_{\text {que }}+\Delta \tau, & \text { if } s_j \text { from } x^{\text {que }} \end{aligned}\right. τsj​​=⎩⎪⎪⎨⎪⎪⎧​τins ​+Δτ,τdems ​,τque ​+Δτ,​ if sj​ from xins  if sj​ from xD if sj​ from xque ​

于是对每一个 segment ，就可以根据其 compression ratio 和 PPL distribution 计算具体的 ppl threshold γj\gamma_jγj​ ，然后根据这个阀值判断保留哪些 token ：

s~j={sj,i∣p(sj,i)>γj}\widetilde{s}_j=\left\{s_{j, i} \mid p\left(s_{j, i}\right)>\gamma_j\right\} sj​={sj,i​∣p(sj,i​)>γj​}

这里作者没有提及如何计算出 ppl threshold γj\gamma_jγj​ ，估计就是直接找 top k 了，满足「压缩后长度/原始长度」等于 compression ratio 。

Distribution Alignment

因为前面计算概率的时候，使用的是一个小模型 Ms\mathcal{M}_sMs​ ，和最终推理使用的大模型可能存在 gap ，这里想去对齐一下。

其实也就是一个蒸馏的过程，让 LLM 根据一些 prefix / instruction 生成 text，然后去训练小模型 Ms\mathcal{M}_sMs​ ：

min⁡θsE[1N∑i=1NL(xi,yi, LLM ;θMs)]\min _{\boldsymbol{\theta}_s} \mathbb{E}\left[\frac{1}{N} \sum_{i=1}^N \mathcal{L}\left(\mathbf{x}_i, \mathbf{y}_{i, \text { LLM }} ; \boldsymbol{\theta}_{\mathcal{M}_s}\right)\right] θs​min​E[N1​i=1∑N​L(xi​,yi, LLM ​;θMs​​)]

(xi,yiLLM)\left(\boldsymbol{x}_i, \boldsymbol{y}_i^{\mathrm{LLM}}\right)(xi​,yiLLM​) : (instruction xi, LLM generated texts yiLLM)\left( \text{instruction}\ \boldsymbol{x}_i \text{, LLM generated texts} \ \boldsymbol{y}_i^{\mathrm{LLM}} \right)(instruction xi​, LLM generated texts yiLLM​) pair

NNN : the number of all examples

LongLLMLingua

Microsoft - LongLLMLingua: ACCELERATING AND ENHANCING LLMS IN LONG CONTEXT SCENARIOS VIA PROMPT COMPRESSION

LongLLMLingua 出发点和LLMLingua就不太一样了，不只是想要压缩prompt保证少掉精度，
而是想要在Long Context Scenarios下，通过提升prompt中的 信息密度 ，从而提升性能。

这个方法特别适合现在普遍使用的Retrieval-Augmented Generation Method。

  1. a question-aware coarse-to-fine compression method
  2. a document re-ordering mechanism to reduce information loss in the middle
  3. dynamic compression ratios
  4. a post-compression subsequence recovery

Question-Aware Coarse-Grained Compression

定义一个 metric rkr_krk​ 给每一个 document xkdoc\mathbf x_k^{\mathrm{doc}}xkdoc​ 打分 ，然后仅保留分数高的 top-n 个 document 。

LLMLingua 使用 document-level PPL：

rk=1/Nk∑iNkp(xk,idoc)log⁡p(xk,idoc),k∈{1,2,⋯ ,K}r_k = 1 / N_k \sum_i^{N_k} p\left(x_{k, i}^{\mathrm{doc}}\right) \log p\left(x_{k, i}^{\mathrm{doc}}\right), k \in\{1,2, \cdots, K\} rk​=1/Nk​i∑Nk​​p(xk,idoc​)logp(xk,idoc​),k∈{1,2,⋯,K}

NkN_kNk​ 为长度

LLMLingua 没有考虑每一个 document 和 instruction & question 的相关性，
这种做法对于 in-context 的 demonstration 来说是可行的，因为我们候选的 demonstration 基本都是符合要求的，
但是对于 RAG 系统来说，不考虑和 instruction ，question 的相关性往往会带来大量的噪音 。

这里可以选择的定义 rkr_krk​ 的方法有：双塔匹配，re-ranker 打分等 …

本文选择 the perplexity of the question $\ \ \mathbf{x}^{\mathrm{que}}\ $ conditioned on document $\ \mathbf{x}^{\mathrm{doc}}\ $ :

rk=1Nc∑iNcp(xique,restrict ∣xkdoc)log⁡p(xique,restrict ∣xkdoc ),k∈{1,2,⋯ ,K}r_k=\frac{1}{N_c} \sum_i^{N_c} p\left(x_i^{\text {que,restrict }} \mid \mathbf{x}_k^{\mathrm{doc}}\right) \log p\left(x_i^{\text {que,restrict }} \mid \mathbf{x}_k^{\text {doc }}\right), k \in\{1,2, \cdots, K\} rk​=Nc​1​i∑Nc​​p(xique,restrict ​∣xkdoc​)logp(xique,restrict ​∣xkdoc ​),k∈{1,2,⋯,K}

这里输入前缀就是每一个 document $\ \mathbf{x}^{\mathrm{doc}}\ $ ，被计算 PPL 的输出则是 a restrictive statement question xique,restrict x_i^{\text {que,restrict }}xique,restrict ​ ：

xque;We can get the answer to this question in the given documents.\mathbf{x}^{\mathrm{que}}; \text{We can get the answer to this question in the given documents.} xque;We can get the answer to this question in the given documents.

可以看成是加了一个正则项，来防止模型的幻觉问题。

NcN_cNc​ 表示 xique,restrict x_i^{\text {que,restrict }}xique,restrict ​ 的 token 数量

Question-Aware Fine-Grained Compression

和 LLMLingua 一样，现在进行 token-level compression ， xins,xque\mathbf{x}^{\mathrm{ins}},\mathbf{x}^{\mathrm{que}}xins,xque 和原本的做法一样是一样的 。

而现在对于 xdoc\mathbf{x}^{\mathrm{doc}}xdoc 部分，作者进行改进，使之感知 question ，用来保留更多和 question 相关的信息 。

作者提出 token-level contrastive perplexity ( the distribution shift caused by the condition of the question ) ：

si=perplexity⁡(xi∣x<i)−perplexity⁡(xi∣xque ,x<i)s_i=\operatorname{perplexity}\left(x_i \mid x_{<i}\right)-\operatorname{perplexity}\left(x_i \mid x^{\text {que }}, x_{<i}\right) si​=perplexity(xi​∣x<i​)−perplexity(xi​∣xque ,x<i​)

然后根据 token-level contrastive perplexity 进行 compression 。

如果该 document 是这个 question 的答案的话，则他的 contrastive perplexity 会很高 。

REDUCE INFORMATION LOSS IN THE MIDDLE

根据 Lost in the Middle ，我们应该将最重要的 doc 放在 context 的头部或者尾部 。

在 Question-Aware Coarse-Grained Compression 之后，我们已经得到了每一个 document 的分数，所以这里直接根据分数从大到小进行排序：

(xins ,x1doc ,⋯ ,xK′doc ,xque )→rk(xins ,xr1doc ,⋯ ,xrK′doc ,xque )\left(\mathbf{x}^{\text {ins }}, \mathbf{x}_1^{\text {doc }}, \cdots, \mathbf{x}_{K^{\prime}}^{\text {doc }}, \mathbf{x}^{\text {que }}\right) \xrightarrow{r_k}\left(\mathbf{x}^{\text {ins }}, \mathbf{x}_{r 1}^{\text {doc }}, \cdots, \mathbf{x}_{r K^{\prime}}^{\text {doc }}, \mathbf{x}^{\text {que }}\right) (xins ,x1doc ​,⋯,xK′doc ​,xque )rk​​(xins ,xr1doc ​,⋯,xrK′doc ​,xque )

ACHIEVE ADAPTIVE GRANULAR CONTROL DURING COMPRESSION

在 LLMLingua 的细粒度 token-level compression 中，所有 document 的压缩比都是相同的 ；但是实际上 document 越重要，他能保留的 token 数量应该是越多的。

所以这里在 token-level compression 时，借助上面每个文档的分数 {rk}k=1K′\{r_k\}_{k=1}^{K'}{rk​}k=1K′​ 计算保留的token数量 / 压缩比。

初始化的压缩比和 LLMLingua 一样：

IMPROVE THE INTEGRITY OF KEY INFORMATION

在 token-level compression 中，可能会存在一种情况： 连续实体中的某个token被删除了 。

作者提出 subsequence recovery ，从最终模型回复中找到一个与最终 compressed prompt 共享的连续子串，在原始 prompt 中匹配，得到真实的结果。

FAQ

  1. 不同LLMs之间Tokenizer 不同会影响压缩效果吗？
    实际上我们用的小模型和大模型之间的Tokenizer完全不同，但是并不影响大模型对compressed prompt的理解。Token-level Compression是为了进一步感知LLMs中对于prompt的token的重要分布。

LLMLingua-2: Data Distillation for Efficient and Faithful Task-Agnostic Prompt Compression

Dataset Construction

主要的数据构造思路还是蒸馏 GPT 4 的信息，需要进行 token-level 的标注，表示这个 token 是否可以被 drop 。

构造数据的三个要求：(i) Token reduction (ii) Informativeness (iii) Faithfulness

如果一次性将所有 prompt 输入给 gpt4，则返回结果往往有很高的压缩率，所以这里首先分 chunk ，长度不超过 512 tokens ，然后单独将每一个 chunk 送给 gpt4 进行压缩。

输入的 prompt 为：

得到被压缩的文本后，就是要进行数据标注了，也就是如何对齐压缩后文本和压缩前文本的每一个token 。

刚开始，所有原始 prompt 中的 token 标签初始化为 False ，然后遍历每一个压缩后文本中的 word ，在原始文本中进行「滑窗模糊匹配」，标注为 True 。

为了进一步控制数据质量，这里引入两个评价指标：
