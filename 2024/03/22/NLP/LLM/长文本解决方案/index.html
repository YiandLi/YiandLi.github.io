长文本解决方案

Posted on 2024-03-22 In NLP , LLM Inference , LLM Model

问题：
算法角度：本身不支持直接外推 -> 位置编码
工程角度： kv cache 太大，prefill 阶段显存太大，计算慢。generation 阶段 kv cache 占用太大，存不完。

通用：
GQA ；位置编码外推 ；滑窗 attention ；LogN-Scaling

关于位置编码长度外推，移步 ： 位置编码1-RoPE,ALiBi和外推方案

工程上：
flash attention + ring attention ; page attention + distributed kv cache ; kv cache offload ; kv cache quantization .

RoPE外推优化——支持192K上下文长度

Mistral

Pre-train 时，就已经考虑到了超长输入的情况。

  • Grouped-query attention (GQA)
  • Sliding Window Attention (SWA) ：只 attend 4,096 个前面的 token
  • Rolling buffer cache 滑动缓存区 ：推理时，KV cache 只需要维护 4,096 个 token
  • Pre-fill and chunking 缓存预处理

Qwen 16 K

-> [post_link NLP/LLM/Qwen] Qwen

训练时使用的长度是 4096 ；采用 training-free 的一些方法，实验时最大长度可以达到 16384 。

  • Dynamic NTK-aware interpolation → YaRN
  • Log N Scaling
  • Window Attention ，不同层的 window size 都不一样

Yi 200 K

-> [post_link NLP/LLM/Yi] Yi

预训练使用了 4096 ，外推后最高 200 K 。

作者认为，模型能够建模比预训练长度(4K)更长的依赖关系是一种内在的能力，基础模型已经具有建模超过4K依赖的能力。可以通过轻量的后训练/微调过程解锁长上下文能力（论文中仅仅为 100 steps）。

不修改模型架构，仍然是全注意力。只是需要有针对性地构造一些长文数据。

同时训练过程经历了两个部分：

  1. Continue Pre-training -》适应长文本
  2. Supervised Fine-tuning -》对齐

元象的方案：XVERSE-Long-256K

Source：https://www.zhihu.com/question/634006127/answer/3364472978

两个阶段：继续预训练的方法可降低训练量，而ABF和NTK可解决模型attention衰减问题。

  • 第一阶段： ABF+继续预训练 ，序列长度从8K拓展到32K

    • 使用20%的预训练数据进行32K的长序列的继续预训练。

    • ABF：将位置编码RoPE（Rotary Position Embedding）的频率从10000修改成500000。

    • 仅使用继续预训练是无法将序列长度提升到256K的，如此长度的预训练无法完成。

  • 第二阶段： NTK+SFT ，序列长度从32K拓展到256K

    • SFT是通过一问一答的方式，使得模型具备了Chat的能力并且强化NTK拓展序列长度的能力。

    • SFT的关键是如何生成训练数据： 团队使用预训练阶段使用的训练数据来构造长序列的SFT训练数据。

      每个文章都首先得到 QA 对，然后混合不同文章，将对应 QA 作为最终的训练目标。

      以构建多文档QA类数据为例：首先，基于自研的XVERSE-65B模型 生成与单个文章有关的高质量问题回答对 ；然后将文章内容混合成目标长度的整段内容，随机选择与其中某个内容匹配的问题回答对；最后将该问题和回答作为整段内容的问题和回答，构成训练的单个样本。通过上述批量化数据生产管线，我们可以得到32K、64K，一直到256K长度的高质量对话数据。

长序列数据

问题：适合短文本的筛选方案可能会导致错误保留一些低质量长文本
以Gopher论文中对文本处理的规则为例：

  • 删除过短和过长的文档
  • 单词平均长度是3-10之间
  • 过多的符号
  • 停用词过多
  • 重复超过一定的比例（长文本天然存在较高重复率

评估方长文本的方案：pjlab - LongWanjuan: Towards Systematic Measurement for Long Text Quality

  1. 连贯性 Coherence ：衡量上文对下文的增益程度。对同一个「response」，如果「long prefix」带来的信息量更多，则表示这个「long prefix」是合理的。

    可以看出来，是首先将长文本进行滑动窗口切片，然后分别去计算每个切片的分数，每次选择窗口内最后 1/4 长度的片段作为 response，long prefix 则为前面 3/4，short prefix 则为 1/2 - 3/4 的文本。

  2. 衔接性 Cohesion ：衡量两句话的紧密程度，比如连词（Connective）和代词（Pronoun）在文本中的密度，文本中相邻两句话的关系。

可以在不同的 预训练domain 上调节 threshold，过高过低都不好。

作者将长文本分成三个类别：

  1. 整体（Halistic）长文本：书箱、论文、报告、小说等
  2. 聚合（Aggregated）长文本：条目列举、话题接近的短文拼接
  3. 杂乱（Chaotic）长文本：乱码、反爬虫等无意义文本

每一种类别都有不同的指标分布：

总体构建流程：

  1. 过滤长文本
  2. 打分数
  3. 根据 threshold 区分上述三种类别
  4. 移除「杂乱（Chaotic）长文本」，混合「整体（Halistic）长文本」和「聚合（Aggregated）长文本」，会对「聚合（Aggregated）长文本」进行上采样。

    Excluding chaotic texts and employing holistic and aggregated texts for training .

长序列训练方法

  1. tokenizer 效率低：1M token 则需要 5min 来 tokenize 。
    原始文本切块并行进行 Tokenize ，但是有问题，比如将一个单词切为两个。

    滑动窗口，每个片段前后冗余 12 个 character，进行拓展，用overlap token id 来对应重叠问题
    作者开源了 paralleltokenizer 。

  2. 长度外推: 位置

    1. 限制注意力范围
      1. 软化词窗口：ALiBi
      2. 平行词窗口：
        • PCW 将 task tokens和每个context平行进行自注意力计算，context之间不做注意力，位置编码大幅减小。
        • NBCE 用朴素贝叶斯解释，并在形式上做出一定的扩展；主要面向 ICL 任务。
      3. 箭形词窗：LM-infinite ；Streaming LLM
    2. RoPe 内插或者NTK：RePoPE，ChunkLLaMA，Self-Extend；NTK，Yarn
