Data

  • 数据来源：中英文的web文档、百科全书、书籍、代码等。
  • 获取手段：从HTML中提取文本，并使用语言识别工具确定语言。
  • 去重处理：精确匹配去重，MinHash & LSH 模糊匹配去重。
  • 删除低质量数据：rule based & machine learning - based method，即使用多个模型去打分，包含语言模型，文本质量评分模型，专门用于检测识别潜在冒犯性或不恰当的内容的模型，还有人工审查。
  • 引入高质量指令数据
  • 最终数据量为 3万亿个token

Tokenizer

  • BPE算法：select the vocabulary cl100k base as our starting point
  • 数字作为单独的Token
  • 词表大小：152K

Model

常规：

  • LayerNorm + RMSNorm

  • SwiGLU FFN ：4 → 8/3

  • FlashAttention

  • RoPE (Rotary Positional Embedding)，计算时候使用 float32 类型

  • context lengths of 2048

额外的改动：

  • Embedding与输出层：千问模型没有将Embedding层和输出层的参数共享，通过增加显存以期换取更优的性能

  • Bias：只在Attention层添加Bias，其他层（dense，norm）去除了Bias，以期有更好的外推性能

长度外推

长度外推性是一个训练和预测的长度不一致的问题:

1、预测的时候用到了没训练过的位置编码（不管绝对还是相对）；

2、预测的时候注意力机制所处理的token数量远超训练时的数量：导致注意力更加分散（或者说注意力的熵变大），从而导致的训练和预测不一致问题

作者的方法 training-free techniques ，也就是不需要去训练 ：

  • Dynamic NTK-aware interpolation → YaRN

  • LogN-Scaling：https://spaces.ac.cn/archives/9444

    根据「推理上下文长度」和「训练上下文长度」，重新缩放 attention 点积的值，确保随着句长的增加，交差熵、点积等值保持稳定。

    softmax⁡(QK⊤d)V ⟹ softmax⁡(log⁡mndQK⊤)V\operatorname{softmax}\left(\dfrac{Q K^{\top}}{\sqrt{d}}\right) V \ \Longrightarrow\ \operatorname{softmax}\left(\dfrac{\log _m n}{\sqrt{d}} Q K^{\top}\right) Vsoftmax(d​QK⊤​)V ⟹ softmax(d​logm​n​QK⊤)V

    mmm 为训练长度，nnn 为预测长度 ；实现时， nnn 为当前 query token 的 id 。

    从下面源码可以看到，这个操作仅推理时使用。用在 Rope 变换之后，key 维度为 [b,s,h] ，query 维度为 [b,1,h] 。

    seq_start:seq_end 等价于 seq_start ，表示 query token 的 id，或者说前缀 token 的长度，正好对应了公式中的 nnn 。

这样做可能仍然有问题，详见 ： leshare : LogN-Scaling 在Alibi下的校正

  • Window Attention：将注意力固定在一个窗口，防止模型关注到句子太远的地方。
    同样观察到：低层 layer 对长度外推更加敏感，因此不同层选择不同的 window size ，底层采用更小的窗口。

Data

数据去重: 归一化后的精确EM去重；MinHash，LSH算法的模糊去重

低质量数据清洗: 基于规则的数据过滤；语言模型打分、文本质量打分模型，用于识别潜在攻击性或潜在攻击性不适当的模型

针对某些数据源进行上采样，以确保模型接受多样化高质量内容的训练

预训练数据加入指令学习的数据

评测数据的去重，防止数据泄漏

Qwen 2

你好，Qwen2
QWEN2 TECHNICAL REPORT

Chat模型都统一更名为Instruct模型

模型结构：

  1. 所有尺寸的模型都使用了 GQA
  2. 针对小模型，由于 embedding 参数量较大，我们使用了 tie embedding 的方法让输入和输出层共享参数，增加非 embedding 参数的占比。
  3. 预训练上下文长度全部是 32K
    通过 YARN 或 Dual Chunk Attention 等技术，Qwen2-7B-Instruct和Qwen2-72B-Instruct均实现了长达128K tokens上下文长度的支持。
  4. 对齐：训练方面，结合了有监督微调、反馈模型训练以及在线DPO等方法。还采用了在线模型合并的方法减少对齐税。

  1. Qwen2-57B-A14B是什么？
    这是一个MoE模型，如果你有关注过之前Qwen1.5的MoE模型，那么你对相关的代码和模型结构应该不太陌生。如果没有，你可以理解为是一个对标Mixtral-8x7B但尺寸和激活参数量都会更大一些的MoE模型。它总共有57B的参数，但是每次前向只激活14B的参数，那么你可以大体认为你能享受14B的推理效率，而它的效果呢是要超过此前的Qwen1.5-32B的。这个模型是奔着性价比去的。伴随而来的新的问题是，这个性价比似乎还没法得到完美的兑现，生态层面存在众多对MoE的支持问题效率问题等。而且57B的总参数量对很多用户来说负担有点太大。但我觉得，随着时间的推移，事情或许会有所改变，很多企业内用户也许能从中挖掘出更大的可能性。

MoE模型的共有72个专家，其中，8个共享专家和64个路由专家，每次模型推理时，8个共享专家一直被使用，路由专家则从64个中选择8个激活。
讨论：“看Qwen2-MOE的代码，应该只有1个共享专家吧？”;“实际上是8个共享专家，因为专家维度一般都要一致，只是实现方式不同。qwen-moe里是利用一个大的向量代替多个小的向量。原理可见deepseek-moe的论文”;“在专家输出加权和的时候，如果8个共享专家的权重是一样的，那8个小共享专家确实等价于1个大共享专家；如果8个权重是不一样的，由共享专家router来学的话，应该就不等价了吧。
作者你写的也没错，是我刚开始理解的时候，下意识以为8个共享专家的权重是不同的”

  1. 为什么有的Tie Embedding有的没有？Tie Embedding指的是embedding和output layer共享参数。对于小模型来说，这非常重要，否则你的non-embedding参数占比过小，非常影响模型效果。但如果对于相对比较大的模型来说，这就比较无所谓，我们也不太必要去省这部分参数。
