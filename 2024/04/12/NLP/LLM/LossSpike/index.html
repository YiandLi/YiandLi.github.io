Some Paper about Loss Spike - 1：Adam

Posted on 2024-04-12 In NLP , LLM Training
Symbols count in article: 2.9k Reading time ≈ 19 mins.

总结一句话，从优化器Adam 的角度考量，原因就是异常 instance 导致的异常 grad norm 。解决方案就是：

  1. 如何发现异常 grad norm
  2. 发现的时候如何处理：
    • 跳过当前样本
    • 不更新当前参数
    • 减小参数更新量：降低学习率，或 直接降低更新量，或 在参数层面上进行滑动一阶平均（GLM130B）。
    • batch: 选用很大的batch size; 动态地增加batch size的策略，GPT3逐渐从32K增加到3.2Mtokens。

The maximum Adam update is proportional to 1/(1−β2+ε)1/(1-β_2 + ε)1/(1−β2​+ε) — and just sweeping β2 was apparently enough to confirm that it does, in fact, have a huge effect on whether we get loss spikes.

基于这个发现，作者提出一种改良的 AdamW （类似 AdaFactor），如果某个参数的梯度的平方过大，则进行 Clip ：

就是首先照常计算包含当前step的「一阶动量」和「二阶动量」；然后做判断，如果当前的「gradient 平方」大于「二阶动量」，则表示这个梯度值有问题，这个时候其实尽量就不要更新这个参数了，这里的做法是让学习率减小，学习率近似乘以 utgt2\dfrac {u_t} {\sqrt{g_t^2}}gt2​​ut​​ 。

在 ImageNet 上微调 ViT，这种方法比「直接对 gradient 进行 clip」效果要好，所以不仅仅能缓解 spike，还能涨点！

每一个 step 上，真实的更新量为 ：

r[i,t]=m[i,t]v[i,t]=u[i,t]∣ε=0r[i, t]=\frac{m[i, t]}{\sqrt{v[i, t]}}=\left.u[i, t]\right|_{\varepsilon=0} r[i,t]=v[i,t]​m[i,t]​=u[i,t]∣ε=0​

在 t=1 时，历史动量为 0，那么更新量也就近似于 gt∣gt∣+ε=±1\dfrac {g_t} {|g_t|+\varepsilon} = \pm 1∣gt​∣+εgt​​=±1 。
在 t 很大时，训练后期，已经逼近了最优点，梯度近似为 0 ，那么更新量为 0 。
也就是理想的训练过程是一个从 1/-1 双峰分布到 0 正态分布的转变，并且因为所有梯度都近似为 0，方差应该不断变小，比如：

但事实并非如此，更新参数会再次进入非稳定的双峰状态，从而导致 loss spike 。

为什么呢？

  1. 一般训练到稳定阶段，浅层的语义知识表示此时一般已经学习的较好，还有梯度消失问题，从而浅层的 grad norm 小于深层的 grad norm，并且不断逼近 0 ，那么也就导致了 ∣gt∣≪ε\left|g_t\right| \ll \varepsilon∣gt​∣≪ε ，之后会导致 ∣mt∣≪ε,vt≪ε0\left|m_t\right| \ll \varepsilon, \quad v_t \ll \varepsilon_0∣mt​∣≪ε,vt​≪ε0​ ，从而浅层网络的更新量近乎为 0 ，得不到更新。而深层网络仍然在进行更新。

    这里和 GLM130B 对不上呀，GLM 发现 “在早期训练中，「嵌入层梯度范数」通常比其他层的梯度范数大几个数量级，并且会大幅波动” ，不过解决方案是差不多的，「step1 发现梯度异常」GLM 直接发现了 Embedding ，「step2 处理」前面paper都是从梯度角度进行规约，GLM是让 Embedding 的参数进行一阶动量更新 。

  2. 长时间的「浅层(embedding层)参数长时间不更新，但是深层的参数依然一直在更新」，会导致突然有一个时间点，不更新的浅层网络无法配合一直更新的深层网络，从而得到比较大的梯度更新。此时 ∣gt∣≫ε,ut≃gt±gt+ε\left|g_t\right| \gg \varepsilon, \quad u_t \simeq \frac{g_t}{ \pm g_t+\varepsilon}∣gt​∣≫ε,ut​≃±gt​+εgt​​ 再次集中在 ±1\pm 1±1 附近 ，变成双峰的非稳定状态，本文提到了浅层(embedding层)这种突然的参数变化可能造成模型的连锁反应进而出现loss spike的现象 。（这也对应了更换样本重新训练有可能会减少loss spike的出现频率，实际上就是选择分布变化较小的样本，减小浅层梯度变换幅度。）

    （可以想象成一个水坝蓄水太久终于被冲开了。至于小模型为什么不会出现这种情况，推测是小模型函数空间小，无法捕获样本的分布变化，越大规模的模型对样本之间不同维度的特征分布变化越敏感）

  3. 这个阶段模型处于非稳态，梯度变化幅度较大，每一次的梯度变化和更新参数变化事件之间又出现了一定的独立性，因此经过一定的时间之后模型有可能再次进入稳态，loss再次drop back down，但不绝对，也就是可能loss发散了。

结论，和 ε\varepsilonε 的大小有关系 ，因为一旦梯度远小于 ε\varepsilonε ，梯度基本就不更新了，时间长了就出现了 loss spike 。

如何防止？

  1. 如之前提到的PaLM和GLM130B提到的出现loss spike后更换batch样本的方法
  2. 减小learning rate，这是个治标不治本的办法，对更新参数的非稳态没有做改进
  3. 减小 ε\varepsilonε 大小
  4. ε\varepsilonε 是否也可以做衰减，随着训练过程逐渐减小

博客汇总

可怕！llm训练的bug，梯度累计设置过大，会导致最终loss过大。

  1. 短序列：上下文短，信息不足，所以不容易预测，loss 偏大（梯度累积再给他加权），于是观测到的现象就是“梯度累积 loss 偏大”。其实它是偏向了短序列的 loss，而不是往大的方向偏。

  2. SFT loss 计算的那些坑（多轮合并/packing）
    SFT中算loss通常来讲都是样本内作token-level mean，样本间作sequence-level mean。如果不同样本间作token-level mean，则会使target token数量多的样本更受重视（相当于被upsample），从而引入不同样本间的不平衡。github.com/THUDM/LongAl

SFT 的 naive 训练方法：每一个 instance 是一个 concat(input_ids, labels)，然后就可以把训练跑起来。
然而，这样的训练效率实际上非常低。通常有两个加速方法：1. 多轮合并 2. packing。
无论是哪种方法，加速后都需要保证 loss 和原来是等价的 ，这也就是最关键的问题！
1. Naive 版本，当前一个 session，有三轮对话，将 chatbot 的每一次输出都作为一个 instance 进行训练：

这样计算的loss可以表示为:
$$
\text { loss }=\frac{1}{3}\left(\frac{l_1}{n_1}+\frac{l_2}{n_2}+\frac{l_3}{n_3}\right)
$$
其中 lil_\mathrm{i}li​ 表示第 i 个 response 的 loss, ni\mathrm{n}_\mathrm{i}ni​ 表示第 i 个 response 输出的 token 数量（对应绿色部分）
是一种 response level avg，这个时候，如果 l3 比较长，l1 比较短，那么其实 l1 的每个 token 分到的 loss 是比较高的（因为分母更小）。
2. Naive 版本存在前缀重复计算的问题，导致训练效率低下。理论上我们可以直接一次行直接前向传播这一整个 session ，即将三轮三个样本合并成一个样本。

理论上看作一个样本，仅计算绿色部分的 token loss 即可，利用 pytroch 的ce loss:
$$
\ell(x, y)= \begin{cases}\sum_{n=1}^N \frac{1}{\sum_{n=1}^N w_{y_n} \cdot 1\left{y_n \neq \text { ignore_index }\right}} l_n, & \text { if reduction = ‘mean’; } \ \sum_{n=1}^N l_n, & \text { if reduction = ‘sum’ }\end{cases}
$$
所以我们会得到这样的 loss:
$$
\text { loss }=\frac{l_1+l_2+l_3}{n_1+n_2+n_3}
$$
即一种 token level avg ，但是这样会导致一个问题，即短 response 分到的 loss 就比之前小了，总体上是弱化短回复的 loss 。
3. 多样本 packing：将所有样本拼接成一条，并且加入 attention mask 保证后面的样本看不见前面的token。

和之前一样，如果不修改 loss 计算方法，packing 的样本之间会存在因为长度不同，导致训练不充分的问题。

正确方法：
一般情况下，loss 计算会经历三次平均：

1. micro batch 维度，分母是这个 micro batch 中的所有 label 不是 -100 的 token 数 （如上所述，有问题）
2. DP 维度，分母是 DP size / GPU数量 ；（其实也存在问题，假设两张卡，一张上一条样本，有两个 packed response ，另一张也是一个样本 有一个 packed response，其实这样平均两个 response 的卡的 loss 是被减少了的）
3. 梯度累加维度，分母是梯度累加数（和 2 同理）

我们这里要做的就是禁用这三个平均，统一用 global batch 的对话轮数作为分母，即 $\frac {\sum \frac{\text{total token loss in this response}}{\text{response token num}}}{\text{response num}}$ ，在新版 megatron 框架中，开启开关 --calculate-per-token-loss 即可禁用 DP 和梯度累加的平均，然后修改 loss_func，每个 micro batch 都需要返回这个 micro batch 的轮数，最后框架会自动将所有轮数求和，作为分母。对于分子，需要除以这个轮次的token 数。正确实现代码如下
