指标

吞吐量1 throughput： query/s 每秒钟可以同时处理多少个 query ，也就是 batch num 。
吞吐量2：推理服务器在所有用户和请求中每秒可生成的 token 数。

延迟 latency： s/token 生成每个 token 所需要的时间，最好能匹配用户的阅读速度，250words/s 。
首个词元生成时间（Time To First Token，简称TTFT）：即用户输入查询后，模型生成第一个输出所需的时间。可能对应 prefilling？
单个输出词元的生成时间（Time Per Output Token，简称TPOT）：为每个查询系统的用户生成一个输出词元所需的时间。也就是上面的 延迟 。

目标：以最短的时间生成首个词元、达到最高吞吐量以及在最短的时间内生成输出词元。

TensorRT-LLM benchmark 部分的几个 performance：

  1. tokens_per_sec : end2end throughput of end to end inference， It is computed by generated_tokens / total_latency.
  2. generation_tokens_per_second : only consider generation. It means the thorughput during generation and computed by generated_tokens / generation_time.

结论

  • 输出长度 决定了整体响应时延：对于平均时延，通常只需将 「预期/最大的输出token长度」 与 模型的每个输出词元的整体平均时间 相乘。
  • 输入长度 对性能来说影响不大，但对硬件要求至关重要：在MPT模型中，「添加512个输入token」增加的时延要少于「生成8个额外输出token」的时延。然而，支持长输入的需求可能使模型难以部署。例如，我们建议使用A100-80GB（或更新版本）来为最大上下文长度为2048个词元来部署MPT-7B模型。
  • 整体时延 与 模型大小 呈次线性关系：在相同的硬件上，较大的模型速度较慢，但速度比不一定与参数数量比相匹配。MPT-30B的时延约为MPT-7B时延的2.5倍，LLaMA2-70B的时延约为LLaMA2-13B时延的2倍。

影响因素

generation 过程中，假设模型参数量为 P，Batch Size 为 B，那么每次生成 token 大约需要 2PB 的 Flops 。需要的内存搬运量为 P 。
P/内存传输速度 为一次性加载模型需要的时间，是一个定值；2PB/flops 为计算时间，随着 B 增加线性增加。
这两个时间存在一个交叉点 B* ，低于 B* 则基本上是在浪费 Flops，为内存受限；高于 B* 则时延开始增加，变成计算受限制。

真实情况下，我们无法达到 B* ，受限于激活值和kv缓存的压力，最大的 B 也会远远低于 B* ，比如如果最大长度为 4k， A100 的最大 batch size 为 35，但是 B* 为 400 。
