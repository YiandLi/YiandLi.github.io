实例化进程

通过 python demo.py 命令开始执行 demo.py 这个程序时，程序被赋予了声明，成为一个进程，这个进程是主进程。
Process() 方法构造的进程都是子进程。

  • 参数：

    • target：调用对象，表示子进程需要调用的方法。
    • args：使用关键字的方式来指定参数，args指定的为传给target函数的位置参数，是一个元组形式，必须有逗号。
    • kwargs：调用对象的字典，传入的是关键字形参，比如kwargs={'name':'egon','age':18}。
    • name：子进程的名称

  • 方法：

    • p.start()：启动进程，并调用该子进程中的 p.run()。
    • p.run()：进程启动时运行的方法，正是它去调用target指定的函数。
    • p.terminate()：强制终止进程p，不会进行任何清理操作（子进程，锁）。
      • 不会立即关闭，所以is_alive立刻查看的结果可能还是存活。
      • 注意：如果p创建了子进程，该子进程就成了僵尸进程，使用该方法需要特别小心这种情况。
      • 如果p还保存了一个锁那么也将不会被释放，进而导致死锁。
    • p.is_alive()：如果p仍然运行，返回True
    • p.join(timeout=None):主线程等待p终止（强调：是主线程处于等的状态，而p是处于运行的状态）。
      • timeout是可选的超时时间，需要强调的是，p.join()只能join住p.start()开启的进程，而不能join住p.run()开启的进程

  • 属性：

    • daemon=False：必须在p.start()之前设置，如果设置为True，代表p为后台运行的守护进程，当p的父进程终止时，p也随之终止；同时p不能创建自己的新进程。
    • name
    • pid

进程启动

有三种启动方式：

  • spawn：子进程仅继承有限的资源。适用于 Unix/Linux 和 Windows，Windows 和 MacOS 的默认方式。
  • fork：子进程会继承父进程中所有的资源，仅适用于。Unix/Linux，Unix/Linux 默认方式。
  • forkserver：创建一个单进程的服务进程，专门用来处理子进程的创建，仅适用于 Unix/Linux。

apply() 与 apply_async()

都是一次执行一个任务，

  • apply() 方法调用后，主进程分配一个任务给子进程，然后主进程会阻塞直到子进程完成，所以不支持并行/异步。
  • 但是 apply_async() 支持异步执行，即提交完之后主进程不等待子进程的完成，直接返回 ApplyResult对象 ，继续进行主进程下面的操作（通常是循环内继续分配任务给其他子进程），因而可以实现并发。

pool.apply_async() 这方法返回的是结果是值的对象，不是值本身，所以要加 get() 方法

虽然 apply_async是非阻塞的，函数返回 multiprocessing.pool.ApplyResult 类型的对象。
可以通过调用该对象的 get() 方法，得到函数返回值，但是这个 get() 方法是阻塞的，即该子进程完成之前，主进程都会阻塞。
所以不要放在循环里，先把 ApplyResult 存起来，然后最后统一得到返回值。

map() 和 map_async()

apply() 可以指定不同的函数，而 map() 将相同的函数应用于许多参数，但是这个方法很耗内存，推荐使用 imap().

和 python 内置的 map 函数使用方法类似，参数为 ： func, iterable[, chunksize=None] ，即传递的参数是 iterable 类型，并且pool.map函数会依次将 iterable 中的每个元素分配给一个进程，进行执行。
apply the same function to each item in an iterable and wait for the results.

两个函数的区别在于，

  • map() 是阻塞主进程的，需要等待 iterable 所有元素处理过后，主进程才继续。
    而 map_async() 不会阻塞主进程。
  • map() 直接返回更改后列表
    map_async() 返回 MapResult 对象，需要调用 .get() 方法得到结果。

从使用上来看，两个函数其实区别不大，因为已经将子进程的分配写入到方法中了，map_async()后面总是需要调用 get() 的，总是需要阻塞的，所以差不多的。

chunksize : int 参数，会将原始的数据分成多个 task，每个 task_size 即为该参数，然后将 task 发送给每个子进程进行处理。
不会保证batch 一定是按照顺序依次传递给子进程的。

为什么比设置为 1 效果更好？：不设置 chunksize 参数，那么 Python 将会默认将每个元素看作一个 task，从而创建一个进程（pool 仅仅是控制了同时有多少进程在跑，不是控制最多创建多少进程的），从而导致创建大量进程和频繁的进程间通信和调度。
创建进程和进程之间的切换调度是 multiprocessing 中的主要开销。

  • 如果 chunksize 设置过大，则进程数目会减少，但是每个进程需要处理更多的数据，可能会导致处理速度下降。
  • 如果 chunksize 设置过小，则会创建更多的进程，从而增加进程之间的通信和调度开销，也会降低程序的性能。

imap默认 chunksize 为1，表示不分割。
map/starmap默认大小为 ceil( task_num /(len(self._pool) * 4) )
默认情况其实效果不会很差，如果需要自己设置，则可以通过一些实验来看查看最优的块大小，范围为 [1, len(iterable) / len(pool._pool)]，指数形式递增(log scale)进行试验。

imap() 和 imap_unordered()

首先两个函数都不支持元组式输入参数，输入形式和 map 一样。

  • 输出方法不是 get() 了，而是使用 for 循环的形式：
  • 返回顺序：
    • imap(),map(),map_async()的返回输出结果顺序和输入相同
    • 而imap_unordered()则为不保证顺序，哪个进程更快结束，就先返回哪个进程的结果
  • 进程开始：
    • iter = pool.imap(fn, data) 一旦生成，无论使不使用结果 iter，多进程计算都会并行开始。
      计算结果会缓存在内存中，所以要注意内存用尽的问题。
    • imap() 和 imap_unordered() 都是不阻塞主进程的，而map() 是阻塞主进程的。
  • 返回效果：
    • 列表级别阻塞：map()/map_async()一定是需要所有元素全都都处理好之后，才能调用返回值。换言之，主进程是一直阻塞(.get()也是一种阻塞)到所有元素都得到返回值之后，才继续进行。
    • 元素级别阻塞：而 imap()/imap_unordered()则是用了一种队列的方法，只要这一次 for 拿出的元素已经就位了，即对应线程已经提交答案了，就不会阻塞，而是拿出元素。如果该答案元素还没有被提交，则会阻塞等待该线程。
      • imap 是按照顺序拿出，进程阻塞
      • imap_unordered 则哪个好了就使用哪个，更快，所以也不强调顺序

可以看到，

  1. 首先是非阻塞的，不是等全都处理完再 for 循环输出的。
  2. imap_unordered 返回不按照顺序的 / map 就是按照顺序的。

报错信息

  • Reason: 'error("'i' format requires -2147483648 <= number <= 2147483647")'
    多进程时候，进程间数据交换是通过pickling，因为处理的文本都比较大，当数据pickled时候超过了istruct 的限制，-2147483648 <= number <= 2147483647
    所以子进程返回的数据无法交换给主进程，直接在子进程内部保存了，就不 merge 了。

进程间通信

与线程不同，进程没有任何共享状态，进程修改的数据，改动仅限于该进程内。
每个子进程享有独立的内存空间，接收进程产生的数据不能马上同步到转发进程中。
